## 一、架构设计原则

| 原则 | 说明 | 决策依据 |
|------|------|---------|
| **双租户隔离** | 领榜与优坐标数据、配置、用户体系完全隔离 | CFM-01 用户确认 |
| **小程序优先** | 前端以微信小程序为主，后台为 Web SPA | 需求明确指定微信小程序 |
| **API-First** | 后端纯 API 服务，前端与后端解耦 | 支持4个小程序+2个后台共享后端服务 |
| **可扩展** | 支持10+版本迭代，模块可独立部署 | 用户要求至少10个版本迭代 |
| **合规优先** | 教师资质审核、支付安全、数据隐私优先 | NFR-03/04/05 |

### 2.2 技术选型

| 层级 | 技术 | 选型理由 |
|------|------|---------|
| **小程序** | 微信原生 + WeUI | 性能最优，组件齐全，微信生态深度集成 |
| **后台 Web** | Vue 3 + Element Plus | 企业级后台成熟方案，组件丰富 |
| **API 网关** | Nginx + OpenResty | 轻量高性能，Lua 脚本灵活扩展 |
| **后端服务** | Java (Spring Boot 3.x) | 企业级成熟生态，团队可招聘性高 |
| **数据库** | MySQL 8.0 | 关系型数据为主，事务支持完善 |
| **缓存** | Redis 7.x | 高频查询（需求大厅/教师列表）加速 |
| **消息队列** | RabbitMQ | 通知/佣金计算等异步任务 |
| **文件存储** | 阿里云 OSS | 图片/视频/资质文件存储 |
| **搜索** | Elasticsearch（V2） | 教师搜索/需求搜索全文检索（MVP 用 MySQL LIKE） |
| **部署** | Docker + 阿里云 ECS | 初期成本可控，后期可迁移 K8s |

## 四、数据架构

### 4.1 数据库设计原则

1. **物理隔离**：领榜和优坐标各使用独立数据库实例，不共享表
2. **共享库独立**：用户、支付、消息、文件、日志使用共享库，通过 tenant_id 区分
3. **软删除**：所有业务表采用逻辑删除（deleted_at 字段）
4. **审计字段**：所有表包含 created_at, updated_at, created_by, updated_by

## 七、安全架构

### 7.1 认证授权

| 场景 | 方案 |
|------|------|
| 小程序登录 | 微信授权 + JWT（7天有效期） |
| 后台登录 | 账号密码 + 图形验证码 + Session |
| API 鉴权 | JWT Token 校验 + 角色/权限中间件 |
| 数据隔离 | 每次请求携带 project 标识，Service 层强制过滤 |

### 7.2 数据安全

| 措施 | 说明 |
|------|------|
| 传输加密 | 全链路 HTTPS |
| 存储加密 | 敏感字段（身份证号、银行卡号）AES-256 加密存储 |
| SQL 防注入 | MyBatis-Plus 参数化查询 + 输入校验 |
| 文件安全 | OSS 私有 Bucket + 签名 URL |
| 日志脱敏 | 日志中敏感字段自动脱敏 |

### 7.3 合规要求

| 要求 | 实现方式 |
|------|---------|
| 教师资质审核 | 上线前必须完成资质审核流程，未审核教师接单受限 |
| 支付安全 | 微信支付官方 SDK，不存储支付密钥 |
| 用户隐私 | 隐私协议 + 数据最必要原则 + 用户可注销 |
| 数据隔离 | 领榜与优坐标数据库物理隔离 |

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| 微信小程序审核周期不可控 | 高 | 版本发布延迟 | 预留审核时间，后台配置控制功能开关 |
| 企微 API 变更 | 低 | 监管功能失效 | 封装适配层，变更时只改适配层 |
| 支付对账差异 | 中 | 财务数据不一致 | 每日自动对账，异常告警 |
| 单体部署性能瓶颈 | 低（MVP） | 响应变慢 | 监控 + Redis 缓存 + V5 拆分微服务 |
