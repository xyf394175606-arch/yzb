# 需求分析：优坐标教师入驻与师资管理闭环

## 1. 背景与范围

本稿单独分析优坐标“教师入驻与师资管理闭环”，覆盖教师申请、后台审核、教师档案、课程绑定、前端展示和授课/评价记录回流。输入来源为 Axure 原型中的 `招募老师列表`、`教师申请审核`、`教师管理`、`教师详情`、`课程管理`、`我的课程表`，以及 v1/v2 需求分析稿。

目标是把老师线索、教师申请审核和正式教师档案之间的关系拆清楚，避免把招募线索、申请审核、教师管理三个不同阶段混成一个页面。本稿只做需求分析，不输出接口、数据库或实现方案。

关键结论：

- 已确认需求：后台存在老师招募线索、教师申请审核、教师管理和教师详情；教师详情展示绑定课程、上课记录、评价记录和教学信息。
- 推测需求：教师申请通过后可能进入教师管理形成教师档案，但原型未明确自动生成规则。
- 待确认需求：招募老师列表是否进入教师申请审核、教师申请字段哪些带入教师档案、老师展示页是否从历史页面回流。
- 暂不纳入范围：历史 `英语伴读老师` 页面和未确认回流的老师展示旧版页面。

## 2. 业务规则

- 小程序或二维码入口可产生老师招募线索，后台 `招募老师列表` 负责跟进线索。
- 教师申请审核页面用于处理教师入驻申请，审核可通过或驳回；驳回时原型显示驳回原因输入项，并标注选择驳回时显示且必填。
- 教师审核通过后是否自动生成教师管理记录待确认；若生成，申请人、手机号、主讲学科、年级、性别、教龄等字段可能带入教师档案。
- `教师管理` 负责正式教师档案维护，包括教师基础资料、形象照、资质证明、个人介绍、主讲学科、教授年级和启用/禁用状态。
- 教师可绑定课程，绑定关系影响课程售卖、课程详情展示和排课授课老师选择。
- `教师详情` 汇总教师绑定课程、上课记录、评价记录和教学信息；上课记录来自排课上课闭环，评价记录来自评价闭环。
- 教师状态为禁用时是否仍可展示、排课、绑定课程，原型未明确，需进入待确认。

## 6. PC/移动端差异

PC 端：

- PC 后台负责招募线索跟进、教师申请审核、教师档案维护、课程绑定和教师详情查看。
- 教师详情作为后台汇总页，承接排课和评价结果。

移动端：

- 小程序可见入口包括邀请老师入驻、老师展示、我的课程表中授课信息等，但当前有效页面未完整证明教师端视图。
- 老师展示页是否从历史页面回流待确认。

历史/暂废：

- `英语伴读老师` 作为历史教师列表参考，不进入当前主链路。
- `原优化界面` 中老师展示相关内容仅作为来源说明。

## 8. 页面与跳转逻辑

```text
小程序/二维码提交老师招募或入驻信息
-> 招募老师列表跟进线索
-> 教师申请审核处理申请
-> 审核通过后进入教师管理建档或补充资料（待确认）
-> 教师管理维护档案和绑定课程
-> 课程售卖和排课选择教师
-> 授课记录回流教师详情
-> 评价记录回流教师详情
```

## 9. 待确认问题

- 教师申请审核通过后是否自动生成教师管理记录。
- 教师申请字段哪些带入教师档案，哪些需后台补充。
- 老师展示页是否从 `原优化界面` 回流到当前小程序。
- 招募老师列表是否只做线索跟进，还是会进入教师申请审核。
- 教师禁用后是否可展示、排课、绑定课程或保留历史记录。
- 是否存在教师端独立课表和授课操作。

## 10. 风险与依赖

- 教师档案依赖招募/审核规则，若自动建档规则不清，会影响后台维护流程。
- 教师绑定课程影响课程售卖和排课；禁用规则不清会导致前端展示与教务排课不一致。
- 上课记录依赖排课上课闭环，评价记录依赖评价闭环。

## 11. 下一步动作

- 业务流程阶段拆分老师招募、教师申请审核、教师建档、绑定课程、授课记录回流流程。
- 信息架构阶段确认教师申请、教师管理、教师详情和前端老师展示的页面关系。
- 交互设计阶段补齐审核弹窗、驳回原因、资质上传、禁用确认、空状态。

## 12. 原型设计输入

- 页面范围：招募老师列表、教师申请审核、教师管理、教师详情、课程管理关联老师、我的课程表教师信息。
- 字段批注：申请资料、审核信息、教师档案、资质证明、绑定课程、上课记录、评价记录。
- 按钮批注：添加跟进、审核、详情、新增、编辑、绑定课程、页签切换。
- 黄色说明区：审核通过是否建档、申请字段带入、禁用规则、老师展示页是否回流。
