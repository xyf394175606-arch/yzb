# 需求分析稿：分销模块专项成型

## 背景

分销模块已经在 k12 项目中进入 MVP，覆盖领榜教育和优坐标两个独立项目。现有文件已确认二级分销、佣金、提现、人工审核、系统外打款等核心口径，也已有创客中心原型接入和纠偏记录。本稿在不修改原型、不输出接口字段的前提下，将分销模块需求成型为可确认、可排序、可验收、可交给下游设计链路使用的需求基础。

## 目标

明确分销模块的业务目标、用户痛点、角色动机、运营约束、商业策略、已确认需求、推测需求、待确认需求、暂不纳入范围、MVP 范围、优先级和验收标准。本稿只输出到需求验收标准，不输出业务流程、页面交互、接口字段、高保真原型或最终 PRD。

## 关键结论

- 分销、提现、佣金已确认进入 MVP，且同时覆盖领榜教育和优坐标。
- 两个项目后台完全独立，分销规则分别设计；不得按跨品牌统一分销中台处理。
- 平台内所有用户均可成为分销员，老师和家长均包含在范围内。
- MVP 支持二级分销，覆盖拉新注册家长、拉新注册老师、销售提成和老师课消提成；课消提成仅一级。
- 提现当前口径为人工审核、系统外打款、不限制提现门槛、按月结算；财务审核，管理员查看/复核，客服无审核权限。
- 本轮仍存在金额和风控类关键待确认项：领榜分佣基数、优坐标默认配置、冻结影响、退款冲正、后台配置范围和风控规则。

## 验收标准

| 需求 | 验收标准 | 不包含内容 |
| --- | --- | --- |
| 两项目分销规则独立 | 后续产物能分别说明领榜教育和优坐标分销规则，不合并为一套共用规则 | 不包含跨品牌统一分销中台 |
| 分销进入 MVP | MVP 清单中包含分销关系、佣金、提现和后台审核 | 不代表所有高级报表和风控都进入 MVP |
| 平台内用户可成为分销员 | 家长、老师均被纳入分销员角色范围 | 不要求本稿定义具体开通交互 |
| 二级分销 | 需求中支持一级、二级团队或关系 | 不支持第三级及以上 |
| 佣金类型 | 需求中覆盖拉新注册家长、拉新注册老师、销售提成、老师课消提成 | 不支持二级课消提成 |
| 领榜默认分佣 | 需求中保留已确认默认比例，并标注分佣基数待确认 | 不在本稿拍板计佣基数 |
| 优坐标按配置 | 需求中不写死优坐标默认比例或金额，标注按后台配置和默认值待确认 | 不编造默认数值 |
| 提现闭环 | 需求中包含提现申请、财务审核、系统外打款和到账状态记录 | 不包含自动打款 |
| 权限边界 | 需求中明确财务审核、管理员查看/复核、客服无审核权限 | 不定义具体后台按钮或权限字段 |
| 用户端创客中心 | 需求中说明团队、收益、提现和成员追踪价值 | 不输出页面布局和跳转逻辑 |
| 后台分销管理 | 需求中说明分销商、规则、佣金、提现、报表方向 | 不输出菜单结构和接口字段 |

## 待确认问题

- 领榜分佣基数口径。
- 优坐标默认佣金比例、注册奖励、课消提成数值和归因有效期。
- 分销商冻结后影响范围。
- 退款、退单、课消撤销时佣金失效、冲正和已提现后的处理方式。
- 后台配置范围和是否需要配置审批。
- 分销风控规则是否进入 MVP。
- 提现时效是否承诺 T+3。

## 风险与依赖

- 金额规则如果未确认，会影响业务流程、数据契约、原型文案和 PRD 验收。
- 优坐标配置如果长期保持“按后台配置”但无默认值，测试和演示只能使用占位样例。
- 冻结和冲正规则不清，会导致异常流程和财务对账缺口。
- 自动打款和多级分销暂不纳入，后续不得在原型或 PRD 中误写为 MVP 能力。
- 创客中心原型材料只作为体验线索，不能替代本稿的需求边界。

## 下一步动作

- 主控向用户汇报本稿中的 P0 待确认项，尤其是领榜分佣基数、优坐标默认配置、冻结和冲正规则。
- 若用户暂时无法确认金额细节，业务流程助手可按“配置待定/人工处理/异常待确认”输出候选流程，但不得闭环为最终规则。
- 用户确认关键金额和权限问题后，再回流业务流程、信息架构、交互设计、数据交互和高保真原型。
