# 需求分析：创客中心分销需求

## 背景

创客中心是家长端、教师端用户查看分销团队、收益和提现状态的用户端入口。前序原型评审中，曾出现真实页面、解释性文案、后台管理内容混在同一页面的问题，导致页面归属和业务判断不清。本稿按新版 `requirement-analysis` 规则重新梳理创客中心需求，重点补齐页面层级、状态说明和原型设计输入，不绘制原型，不定义接口、数据库或组件实现。

## 目标

明确创客中心分销需求的业务规则、字段来源、功能按钮、PC/移动端差异、页面层级、状态说明和页面跳转，为后续 Axure 风格 HTML 原型提供可直接使用的输入。

## 关键结论

- 平台口径为“人人都是分销员”，家长和老师都可拉新注册、买课，后续对接排课系统后可体现上课贡献；但后台可以控制具体用户是否开通分销权限。
- 家长端和教师端个人中心入口位置、入口文案一致。未开通用户也要看到创客中心入口，并能申请开通。
- 分销为二级分销，前端统一使用“一星团队、二星团队”；二星团队通过一星成员下钻或树形分组展示，不展示超出二级的团队。
- 创客中心移动端主路径为：个人中心 -> 创客中心入口 -> 创客中心首页 -> 团队成员列表 -> 成员详情页；提现路径为：创客中心首页/团队页 -> 提现中心 -> 提现申请/提现详情。
- 申请开通分销权限后展示等待审核状态；被拒绝时展示具体拒绝原因，并允许重新申请。等待审核、被拒绝、空状态、按钮置灰都属于页面状态，不进入页面层级树。
- 成员贡献金额按后台配置的分成比例展示，来源区分注册、买课、上课等；上课贡献以后续排课系统对接为前提。
- 提现申请采用银行卡 + 登录账号短信验证，采集开户银行、开户人姓名、银行卡号，不做实名校验。
- 提现时效不做承诺。用户提交申请后，后台收到申请；财务在系统外打款，打款成功后在系统内标记打款成功。
- PC 后台需要支撑提现申请、分销比例配置、用户列表-团队列表模块，操作角色由系统自定义权限人工配置，不在需求稿中写死固定角色。
- 分销比例配置需要区分注册、买课、上课来源，以及一星/二星层级比例，并支持生效时间、停用状态和历史版本记录。
- 原型中的解释性文案可以和页面同屏，但必须用标签、红字批注、黄色说明区或便签明显区分，不能伪装成真实产品 UI。

## 1. 背景与范围

本次需求面向平台内家长和老师，解决创客中心分销权限、团队查看、成员贡献、收益提现和后台支撑配置不清的问题。移动端包含个人中心入口、创客中心入口、创客中心首页、团队成员列表、成员详情页、提现中心、提现申请和提现详情页；PC 后台包含提现申请、分销比例配置、用户列表-团队列表。不包含自动打款闭环、超出二级的多级分销、跨品牌统一分销中台、接口设计和数据库设计。

## 风险与依赖

- 若后续原型把后台配置、报表说明或流程说明放进移动端创客中心真实 UI，会再次造成页面归属错误。
- 若把状态当成页面进入导航层级，后续 Axure 页面树会膨胀并影响评审判断。
- 成员贡献金额依赖分销比例配置，分销比例配置页必须支持来源、层级、生效时间、停用状态和历史版本记录。
- 上课贡献依赖排课系统对接，当前只能作为后续接入来源说明，不能在用户端承诺已产生上课分润。
