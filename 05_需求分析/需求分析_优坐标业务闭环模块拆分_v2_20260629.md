# 需求分析

## 1. 背景与范围

本稿基于 `05_需求分析/需求分析_优坐标Axure原型拆分_v1_20260629.md` 进行结构重组，目标是从“页面/菜单视角”调整为“业务闭环模块视角”。每个模块按一个完整业务闭环拆分，并同时标明前端小程序、PC 后台、涉及角色、关键数据、状态流转、页面归属和待确认边界。

本次只处理优坐标，不扩展领榜教育；不覆盖 v1 页面拆分稿；不输出 API、数据库、开发实现方案；不重画原型。Axure 原型中可见的页面、字段、按钮和黄色说明视为原型可见事实；业务状态、字段来源、权限、支付、提现、分佣等无法由原型证明的内容标记为待确认。

关键结论：

- 优坐标应按业务闭环拆分模块，不应仅按后台菜单或小程序页面拆分。
- 一个页面可归属多个业务闭环，例如 `创客中心` 同时涉及创客分佣闭环和提现结算闭环。
- 一个后台页面也可支撑多个闭环，例如 `会员管理` 同时支撑会员权益闭环和渠道商名额闭环。
- 当前可拆为 10 个业务闭环：课程售卖、会员权益、渠道商名额、教师入驻与师资管理、排课上课、评价、创客分佣、提现结算、客户/家长跟进、招募与合伙人线索。
- 旧版/暂废页面只作为来源说明或历史参考，不进入当前业务闭环主链路。

## 2. 业务规则

### 2.1 业务闭环划分规则

- 一个业务闭环必须包含触发入口、前端用户动作、后台管理动作、关键数据、状态变化和闭环结束条件。
- 页面不是模块边界，业务结果才是模块边界；同一页面可被多个闭环复用。
- 前端指小程序用户侧和渠道商端用户侧可见页面、入口和操作。
- 后台指 PC 管理端的配置、审核、处理、查询、记录和财务操作。
- 状态只作为业务状态描述，不放进导航层级。
- 字段必须有页面和来源；无法确认来源的字段进入待确认问题。
- 按钮必须写明操作对象、结果、跳转/弹窗、确认/取消结果；无法确认结果的操作进入待确认问题。

### 2.3 通用状态规则

- 订单状态、会员状态、排课状态、评价状态、分佣状态、提现状态不能进入导航层级，只在对应业务闭环中描述。
- 暂废页、历史页、带日期版本页只作为证据来源，不作为当前业务闭环主链路。
- 小程序 `我的` 页面承载多个业务入口，后续信息架构阶段需要拆分为课程、会员、订单、课程表、创客中心等入口。
- PC 后台菜单可按管理职责组织，但需求分析阶段必须说明其支撑的业务闭环。

## 5. PC/移动端差异

PC 端：

- PC 后台不是单独业务闭环，而是承接配置、审核、处理、查询和记录能力。
- 课程管理、教师管理、会员管理、渠道商管理、分佣比例设置等页面偏配置和主档维护。
- 购课订单、会员充值订单、分佣账单、提现申请、家长查看详情页等页面偏记录查询和处理。
- 一对一、试听课、班课管理、签到偏教务执行闭环。
- 后台页面可能同时支撑多个闭环，后续信息架构不能只按页面名判断业务归属。

移动端：

- 小程序页面是用户触发闭环和查看结果的主要入口。
- `我的` 承载课程、会员、订单、课程表、创客中心等多个入口，需要在后续信息架构拆分。
- `创客中心` 同时承载分佣展示、团队管理、提现申请和提现记录。
- `我的会员权益` 只展示会员权益结果和续费/升级入口，不承担后台权益配置。
- `我的课程表` 和 `我的订单` 同时承担评价入口，需要和评价闭环联动。

历史/暂废：

- `原优化界面` 可作为课程展示、课程详情、会员支付、招商加盟等前端参考，但不默认进入当前闭环主链路。
- `购课用户列表（暂时不用）` 不进入当前闭环。
- 带日期版本页面只作为版本差异来源，例如申请合伙人列表、会员管理、我的会员权益等。

后台创建渠道商 -> 后台充值会员名额 -> 渠道商端开通/续费家长会员或转让名额 -> 生成消耗/转让/充值记录 -> 会员管理同步权益包

提现结算闭环
创客中心立即提现 -> 后台提现申请待结算 -> 后台结算上传凭证或作废 -> 小程序提现记录展示状态和凭证

客户/家长跟进闭环
家长注册/购课/开通会员/产生推荐关系 -> 后台家长管理跟进 -> 家长详情汇总订单/上课/推荐/评价 -> 支撑运营继续跟进

招募与合伙人线索闭环
小程序招募/招商入口提交 -> 后台线索列表 -> 添加跟进 -> 关闭或转为渠道商 -> 渠道商名额闭环

## 8. 待确认问题

### 8.1 课程售卖闭环

- 当前课程详情、课程支付、会员/非会员支付页面是否从 `原优化界面` 回流到当前小程序范围。
- 购课订单的待支付、已支付、已取消、已完成状态流转和支付超时规则。
- 试听课订单是否始终 0 元且默认已完成，是否存在取消或过期。

### 8.2 会员权益闭环

- 普通会员 `499元/年`、卓越会员 `1000元/年` 是否为正式价格。
- 普通会员续费后折扣课时是恢复固定课时、叠加剩余课时，还是覆盖原剩余课时。
- 卓越会员是否仍需要后台赠送折扣课时，还是仅按有效期判断。
- 卓越会员购买普通会员时的阻断提示是否正式采用原型文案。

### 8.3 渠道商名额闭环

- 渠道商端开通/续费会员时，验证码发送给家长账号还是渠道商账号。
- 渠道商关闭后，已开通会员、剩余名额、充值记录、消耗记录是否继续可见或生效。
- 渠道商开通会员时是否可选择普通会员/卓越会员，或固定开通某一权益包。

### 8.4 教师入驻与师资管理闭环

- 教师申请审核通过后是否自动生成教师管理记录。
- 教师申请字段哪些带入教师档案，哪些需后台补充。
- 老师展示页是否从 `原优化界面` 回流到当前小程序。
- 招募老师列表是否只做线索跟进，还是会进入教师申请审核。

### 8.5 排课上课闭环

- ClassIn 是否为排课强依赖；接口失败时是否允许后台手工保存课节。
- 距离开课 5 分钟内不可编辑/删除规则是否适用于一对一、试听课、班课全部场景。
- 签到提交后是否绝对不可修改，还是管理员可补签/撤回。
- 移出班级学生时剩余课时如何处理。

### 8.6 评价闭环

- 课程评价和教师评价开放条件是订单完成、课节完成、签到完成，还是支付完成。
- 一个课程中的一个老师只能评价一次是否作为正式规则。
- 后台删除评价后，小程序评价详情是否隐藏，是否保留删除原因。

### 8.7 创客分佣闭环

- 注册拉新、课程分享成交、教师课消三套分佣链路是否全部进入当前版本。
- 分佣比例和固定金额以后台配置为准，还是需要财务确认后固化。
- 家长和教师是否隐藏二星团员，还是置灰展示并提示不可产生收益。
- 团队动态是否展示访问记录；原型建议后续通过消息推送关键行为。

### 8.8 提现结算闭环

- 提现状态命名统一为待结算/已结算，还是待打款/已打款。
- 最低提现金额、手续费/费率、银行卡校验规则。
- 作废是否需要原因，作废后用户端是否展示。
- 结算凭证哪些角色可查看。

### 8.9 客户/家长跟进闭环

- 账号管理与家长管理边界：账号管理是否只读，家长管理是否负责编辑和跟进。
- 家长关闭后是否影响会员权益、订单查看、分佣归属。
- 跟进记录是否需要下次跟进时间提醒。

### 8.10 招募与合伙人线索闭环

- 申请合伙人列表是否继续沿用 `2026_6_5` 版本。
- 招募合伙人、招商加盟、邀请老师入驻分别进入哪个后台列表。
- 转为渠道商后是否自动启用渠道商账号，是否需要初始名额。

## 9. 原型设计输入

### 9.1 原型范围输入

后续原型设计不应按原后台菜单直接平铺，而应先按业务闭环设定页面分组：

- 前端小程序优先分组：课程购买、会员权益、我的订单、我的课程表、创客中心、招募/招商入口。
- PC 后台优先分组：课程与师资配置、会员与渠道商、订单与财务、排课与签到、评价与客户跟进、分佣与提现。
- 历史/暂废页面作为黄色说明来源，不进入当前主导航。

### 9.2 信息架构输入

- 一个业务闭环可对应多个前端页面和后台页面，IA 阶段需要建立“业务闭环 -> 页面 -> 状态”的三层关系。
- `创客中心` 必须拆成分佣展示和提现申请两个业务能力，不应只作为一个页面处理。
- `会员管理` 必须同时标注支撑会员权益和渠道商名额两个闭环。
- `课程管理` 同时支撑课程售卖、教师绑定和评价展示。
- `家长管理` 同时支撑客户跟进、推荐关系、分佣归属和会员/订单关联。

### 9.3 交互设计输入

- 所有闭环都需补齐空状态、加载状态、错误状态、无权限状态和二次确认。
- 涉及资金或权益的操作必须有确认弹窗：会员开通/续费、名额转让、赠送课时、提现结算、作废。
- 排课和签到必须明确开课时间限制、提交后不可修改、请假不扣课时。
- 评价必须明确可评价、已评价、删除后状态。

### 9.4 数据交互输入

- 虽然本稿不输出接口，但后续数据交互阶段需按闭环拆字段，不按页面孤立拆字段。
- 核心状态字段包括：订单状态、会员状态、名额余额、教师审核状态、课节状态、签到状态、评价状态、分佣状态、提现状态、家长账号状态、线索状态。
- 核心跨闭环关联包括：课程与订单、订单与排课、排课与签到、签到与课时、会员与折扣课时、渠道商与会员名额、推荐关系与分佣、分佣与提现。

### 9.5 黄色说明区与待确认标注

后续原型中的黄色说明区应按业务闭环放置：

- 课程售卖：会员/非会员计价、试听课 0 元、订单仅详情。
- 会员权益：普通/卓越差异、到期/课时用尽、续费/升级规则。
- 渠道商名额：名额充值、开通/续费、转让、记录同步。
- 排课上课：ClassIn、5 分钟限制、签到扣课时。
- 创客分佣：三套独立链路、二级收益限制、师资分佣。
- 提现结算：状态、凭证、作废、用户端展示。

## 风险与依赖

- 如果继续按页面菜单拆分，会掩盖页面复用和业务闭环跨端关系，尤其是会员、渠道商、创客、提现。
- 会员、渠道商、分佣、提现涉及资金和权益，必须由业务、财务、运营确认。
- 排课依赖 ClassIn，需确认外部系统可用性和失败降级。
- 历史/暂废页面内容较多，若不先标记来源，容易误入当前版本。
