# 需求分析：优坐标提现结算闭环

## 1. 背景与范围

本稿单独分析优坐标“提现结算闭环”，覆盖用户申请提现、后台提现申请、结算/作废、凭证查看和用户端提现记录。输入来源为 Axure 原型中的 `创客中心`、`提现申请`，以及 v1/v2 需求分析稿。

目标是把提现从创客分佣中拆出来，单独处理资金申请、后台结算、凭证和状态。创客收益如何产生归属创客分佣闭环，本稿只处理收益进入可提现后的申请与处理。本稿不输出接口、数据库或开发实现。

关键结论：

- 已确认需求：后台 `提现申请` 状态包括待结算、已结算、已作废；待结算可结算或作废；已结算可查看凭证；结算需上传最多 3 张凭证。
- 推测需求：小程序创客中心可查看历史提现记录和打款凭证，但详情页形态未明确。
- 待确认需求：状态命名待结算/已结算与待打款/已打款是否统一、最低提现金额、费率、作废原因、凭证查看权限。
- 暂不纳入范围：佣金生成规则，归属创客分佣闭环。

## 2. 业务规则

- 用户在创客中心有可提现金额后，可点击立即提现发起提现申请。
- 后台 `提现申请` 接收用户提现申请，展示提现人、申请时间、提现金额、收款人、银行卡号、开户行、费率、状态和打款时间。
- 后台状态包括待结算、已结算、已作废。待结算可结算或作废；已结算可查看结算凭证；已作废不能再做其他操作。
- 结算需上传结算凭证，最多支持 3 张图片；打款时间默认为当前时间，可编辑，精确到年月日时分。
- 作废是否需要原因、作废后用户端是否展示、可提现金额是否回退，原型未明确。
- 小程序端提现记录应展示提现状态、金额、申请时间、打款时间和凭证信息，具体页面形态待确认。

## 6. PC/移动端差异

PC 端：

- 后台财务负责查询提现申请、结算、作废、上传凭证和查看凭证。
- 资金状态处理以后台为准。

移动端：

- 小程序创客中心负责用户发起提现、查看提现记录和查看处理结果。
- 小程序是否展示作废原因、手续费和到账时效待确认。

历史/暂废：

- 暂无历史提现页面进入当前主链路。

## 8. 页面与跳转逻辑

```text
创客中心展示可提现金额
-> 用户点击立即提现并提交提现申请
-> 后台提现申请生成待结算记录
-> 财务结算上传凭证，状态变为已结算
-> 或财务作废，状态变为已作废
-> 小程序提现记录展示状态和凭证
```

## 9. 待确认问题

- 提现状态命名统一为待结算/已结算，还是待打款/已打款。
- 最低提现金额、手续费/费率、银行卡校验规则。
- 作废是否需要原因，作废后用户端是否展示。
- 作废后可提现金额是否退回。
- 结算凭证哪些角色可查看，用户端是否可查看完整凭证。
- 小程序提现记录是列表展开、详情页，还是弹窗。

## 10. 风险与依赖

- 提现依赖创客分佣闭环提供准确可提现余额。
- 提现涉及资金和凭证，状态命名、作废和手续费未确认会影响财务风险。
- 如果小程序提现记录形态不明确，用户无法追踪提现处理结果。

## 11. 下一步动作

- 业务流程阶段画提现申请、后台结算、作废、凭证查看流程。
- 信息架构阶段确认小程序提现记录和提现详情页面形态。
- 交互设计阶段补齐最低金额、银行卡校验、上传失败、作废原因、凭证预览。
