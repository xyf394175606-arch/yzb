# 需求分析：优坐标评价闭环

## 1. 背景与范围

本稿单独分析优坐标“评价闭环”，覆盖课程/教师可评价条件、用户提交评价、用户端评价状态更新、后台查看/删除和评价记录回流。输入来源为 Axure 原型中的 `我的课程表`、`我的订单`、`评价管理`、`课程管理`、`教师详情`、`家长查看详情页`，以及 v1/v2 需求分析稿。

目标是把评价从用户动作到后台管理再到详情页展示的链路独立出来，避免在课程售卖、排课和教师管理文档中分散处理。本稿只做需求分析，不输出接口、数据库或实现方案。

关键结论：

- 已确认需求：后台 `评价管理` 区分课程评价和教师评价；小程序 `我的课程表`、`我的订单` 中可发起评价、查看评价详情和删除评价。
- 推测需求：订单或课节达到某状态后开放评价，但具体条件未明确。
- 待确认需求：课程评价和教师评价开放条件、删除后前端状态、一个课程中的一个老师只能评价一次是否正式约束。
- 暂不纳入范围：评价推荐算法、评价审核机制、评价展示排序规则。

## 2. 业务规则

- 评价分为课程评价和教师评价，后台 `评价管理` 可按页签或类型查看。
- 小程序 `我的订单` 可对订单课程发起评价；`我的课程表` 可对课程或教师发起评价。
- 评价提交后，用户侧应从未评价变为已评价，并可查看评价详情；原型中出现删除评价入口。
- 后台可查看评价记录并删除；删除后小程序端展示规则未明确。
- 教师评价进入教师详情评价记录，课程评价可进入课程管理或评价管理记录，家长详情页可展示评价记录。
- 原型说明中出现“一个课程中的一个老师只能评价一次”，是否作为正式规则需确认。

## 6. PC/移动端差异

PC 端：

- 后台负责评价查询、展示、删除和在教师/家长/课程详情中的记录查看。
- 后台不承担用户填写评价。

移动端：

- 小程序负责评价提交、评价详情、评价状态展示和删除评价入口。
- `我的订单` 偏课程/订单评价，`我的课程表` 偏课节/教师评价。

历史/暂废：

- 暂无历史评价页面进入当前主链路。

## 8. 页面与跳转逻辑

```text
订单/课节达到可评价条件
-> 我的订单或我的课程表展示评价入口
-> 用户提交课程评价或教师评价
-> 前端状态变为已评价
-> 后台评价管理生成评价记录
-> 评价记录回流课程管理、教师详情、家长查看详情页
-> 后台或用户删除评价
-> 前端和后台评价状态更新（规则待确认）
```

## 9. 待确认问题

- 课程评价和教师评价开放条件是订单完成、课节完成、签到完成，还是支付完成。
- 一个课程中的一个老师只能评价一次是否作为正式规则。
- 后台删除评价后，小程序评价详情是否隐藏，是否保留删除原因。
- 用户端删除评价是否同步删除后台记录，还是仅隐藏用户侧展示。
- 评价是否需要审核后展示，还是提交后立即展示。

## 10. 风险与依赖

- 评价闭环依赖课程售卖订单状态和排课签到状态。
- 教师评价会影响教师详情和师资展示，如果删除规则不清，前后端展示容易不一致。
- 评价开放条件不清会影响小程序按钮显示和用户体验。

## 11. 下一步动作

- 业务流程阶段拆出订单评价、课节评价、教师评价、后台删除评价流程。
- 信息架构阶段明确评价表单、评价详情是否为独立页面或弹窗。
- 交互设计阶段补齐可评价、已评价、删除确认、删除后提示和无评价空态。

## 12. 原型设计输入

- 页面范围：我的订单、我的课程表、评价管理、教师详情评价记录、家长详情评价记录、课程管理评价区域。
- 字段批注：评价内容、评分、评价人、评价时间、评价状态、删除状态。
- 按钮批注：评价、教师评价、发表、查看评价详情、删除评价、后台删除。
- 黄色说明区：评价开放条件、一个课程老师评价一次、删除后展示规则待确认。
