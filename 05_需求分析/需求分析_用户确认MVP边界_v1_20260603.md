# 用户确认 MVP 边界

## 背景

本文件记录用户在 02 需求分析关键关口对领榜教育、优坐标、后台边界、业务模式、分销提现、佣金和排课能力的确认。该文件作为 `03 - 业务流程助手` 的正式输入之一。

将用户确认内容沉淀为可交给业务流程设计的需求边界，解除 02 需求分析关口阻塞，避免后续助手依赖聊天记忆。

## 关键结论

- 领榜教育后台与优坐标后台完全独立。
- 领榜教育为 C2C 模式：家长发布需求单，老师抢单。
- 领榜教育老师抢单后不需要家长在系统内确认老师；老师付费购买/接单后，需求单进入锁定状态，客服在线下/系统外拉企业微信群，将接单老师拉入群内与家长沟通，平台通过企业微信进行监管。
- 领榜教育同一需求单在未被老师付费购买/接单前允许多个老师查看或尝试抢单；一旦有老师付费购买/接单成功，订单状态变更并禁止其他老师继续抢单。若后续未成交或老师退单，需求单重新开放给其他老师查看和抢单。
- 优坐标为 B2C 模式：家长可以按老师购买该老师关联产品，也可以直接购买产品。
- 优坐标由客服与家长约定上课时间，并由客服选择上课老师；优先选择家长指定老师，也允许客服替换其他老师，后台必须支持该操控能力。
- 优坐标客服替换家长指定老师时，确认动作发生在系统外，由客服与家长沟通确认；系统内不要求家长二次确认。
- 优坐标包含会员卡购买业务。
- 优坐标会员卡先作为独立购买与权益开通处理，当前不做复杂抵扣。
- 领榜教育和优坐标都包含分销业务，但两者分销规则存在差异，需要分别建模。
- 分销、提现、佣金进入当前 MVP。
- 分销规则需支持 2 级分销、拉新注册家长或老师、销售提成、拉新注册老师上课后的课消提成；课消提成只支持一级。
- 第三方排课系统对接方式暂不确定，但当前设计必须包含最小排课业务，确保整体业务可跑通。

## 业务模式确认

| 项目 | 业务模式 | 核心流程方向 |
| --- | --- | --- |
| 领榜教育 | C2C | 家长发布需求单，老师付费购买/接单后锁定需求单，客服拉企业微信群，平台在企微中监管沟通与成交 |
| 优坐标 | B2C | 家长购买老师关联产品或直接购买产品，客服约课并分配老师 |

## 后台边界确认

| 后台 | 边界 |
| --- | --- |
| 领榜教育后台 | 完全独立管理领榜教育的家长需求、老师抢单、分销、提现、佣金等业务 |
| 优坐标后台 | 完全独立管理优坐标的产品、老师关联产品、会员卡、客服排课、老师分配、分销、提现、佣金等业务 |
| 共用后台/共享中台 | 当前不纳入，两个后台完全独立 |

## 下游业务流程关注点

- 领榜教育需要建模：需求单发布、老师付费购买/接单、需求单锁定、系统外企业微信群沟通监管、未成交/退单后重新开放、分销佣金、提现。
- 优坐标需要建模：老师关联产品购买、直接产品购买、会员卡购买、客服约课、客服分配老师、老师替换、系统外家长确认、最小排课、分销佣金、提现。
- 两个后台完全独立，业务流程图、权限边界、状态流转必须分别输出。
- 分销、提现、佣金进入 MVP，但两个项目规则不同，需要分开建模，不得合并为一套共用流程。
- 第三方排课系统暂不确定，流程设计必须先保证不接第三方也能完成最小排课闭环。

## 待确认问题

- 领榜教育企业微信群是否需要在系统内记录群号、群链接、建群时间、进群老师和监管状态。
- 领榜教育老师付费购买/接单后的退单条件、退款规则、重新开放时间和人工审核规则。
- 分销提现门槛、提现审核角色、结算周期和打款方式。
- 2 级分销中一级/二级的具体比例或金额。
- 拉新注册家长、拉新注册老师、销售提成、课消提成的触发条件和归因有效期。
- 优坐标会员卡后续是否需要与产品订单、课时或折扣建立抵扣关系。

## 风险与依赖

- 分销、提现、佣金进入 MVP 后，财务规则和状态流转复杂度会上升。
- 第三方排课系统不确定时，必须避免流程强依赖第三方接口。
- 两个后台完全独立后，后续不得再按共享中台或共用后台设计。
- 领榜核心服务沟通在系统外企业微信完成，系统内需至少保留可追溯的监管记录，否则后续客服管理和纠纷处理会缺少依据。
