# 用户确认 MVP 边界

## 背景

本文件记录用户在 02 需求分析关键关口对领榜教育、优坐标、后台边界、业务模式、分销提现、佣金和排课能力的确认。该文件作为 `03 - 业务流程助手` 的正式输入之一。

## 目标

将用户确认内容沉淀为可交给业务流程设计的需求边界，解除 02 需求分析关口阻塞，避免后续助手依赖聊天记忆。

## 输入来源

- 用户在主控对话中的确认
- `05_需求分析/需求分析_领榜优坐标MVP范围_v1_20260603.md`
- `04_需求澄清/需求澄清_领榜优坐标问题清单_v1_20260603.md`

## 关键结论

- 领榜教育后台与优坐标后台完全独立。
- 领榜教育为 C2C 模式：家长发布需求单，老师抢单。
- 优坐标为 B2C 模式：家长可以按老师购买该老师关联产品，也可以直接购买产品。
- 优坐标由客服与家长约定上课时间，并由客服选择上课老师；优先选择家长指定老师，也允许客服替换其他老师，后台必须支持该操控能力。
- 优坐标包含会员卡购买业务。
- 领榜教育和优坐标都包含分销业务，但两者分销规则存在差异，需要分别建模。
- 分销、提现、佣金进入当前 MVP。
- 第三方排课系统对接方式暂不确定，但当前设计必须包含最小排课业务，确保整体业务可跑通。

## 业务模式确认

| 项目 | 业务模式 | 核心流程方向 |
| --- | --- | --- |
| 领榜教育 | C2C | 家长发布需求单，老师抢单，后续完成撮合与服务 |
| 优坐标 | B2C | 家长购买老师关联产品或直接购买产品，客服约课并分配老师 |

## 后台边界确认

| 后台 | 边界 |
| --- | --- |
| 领榜教育后台 | 完全独立管理领榜教育的家长需求、老师抢单、分销、提现、佣金等业务 |
| 优坐标后台 | 完全独立管理优坐标的产品、老师关联产品、会员卡、客服排课、老师分配、分销、提现、佣金等业务 |
| 共用后台/共享中台 | 当前不纳入，两个后台完全独立 |

## MVP 范围确认

| 能力 | 是否进入 MVP | 说明 |
| --- | --- | --- |
| 领榜需求单 | 是 | 家长发布需求单 |
| 领榜老师抢单 | 是 | 老师端抢单，后台可管理 |
| 优坐标老师关联产品购买 | 是 | 家长选老师并购买关联产品 |
| 优坐标直接购买产品 | 是 | 家长不选老师也可购买产品 |
| 优坐标会员卡购买 | 是 | 会员卡业务进入 MVP |
| 优坐标客服约课 | 是 | 客服与家长约定上课时间 |
| 优坐标客服分配老师 | 是 | 优先指定老师，可替换老师 |
| 最小排课业务 | 是 | 不依赖第三方系统也要跑通 |

| 提现 | 是 | 与分销佣金一起进入 MVP |
| 佣金 | 是 | 需要区分两个项目规则 |
| 第三方排课深度对接 | 待确认 | 当前不作为强依赖，先做最小排课闭环 |

## 下游业务流程关注点

- 领榜教育需要建模：需求单发布、老师抢单、抢单状态、家长/平台后续选择或确认、分销佣金、提现。
- 优坐标需要建模：老师关联产品购买、直接产品购买、会员卡购买、客服约课、客服分配老师、老师替换、最小排课、分销佣金、提现。
- 两个后台完全独立，业务流程图、权限边界、状态流转必须分别输出。
- 分销、提现、佣金进入 MVP，但两个项目规则不同，需要分开建模，不得合并为一套共用流程。
- 第三方排课系统暂不确定，流程设计必须先保证不接第三方也能完成最小排课闭环。

## 待确认问题

- 领榜教育老师抢单后是否需要家长确认老师，还是抢单成功即进入后续服务。
- 领榜教育需求单是否允许多个老师同时抢单。
- 优坐标客服替换老师时是否需要家长二次确认。
- 优坐标会员卡和产品购买之间是否存在权益抵扣、课时抵扣或折扣关系。
- 领榜教育和优坐标的分销佣金计算规则、提现门槛、审核角色和结算周期。

## 风险与依赖

- 分销、提现、佣金进入 MVP 后，财务规则和状态流转复杂度会上升。
- 第三方排课系统不确定时，必须避免流程强依赖第三方接口。
- 两个后台完全独立后，后续不得再按共享中台或共用后台设计。
