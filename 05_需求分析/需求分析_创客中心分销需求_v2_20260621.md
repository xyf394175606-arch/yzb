# 需求分析：创客中心分销需求

## 背景

创客中心是分销员在用户端查看团队、收益和提现状态的入口。当前分销、佣金、提现已进入 MVP，但原型评审中发现“页面画面、评审说明、后台内容”容易混在一起，影响需求判断。本稿只分析创客中心需求，作为后续原型设计输入，不绘制原型，不定义接口、数据库或组件实现。

## 目标

明确创客中心的业务规则、字段来源、按钮动作、PC/移动端差异、页面跳转和待确认问题。重点约束：用户端创客中心默认首屏只展示用户真实可见内容；解释性文案可以同屏出现，但必须以说明标签或便签形式与产品 UI 明显区分。

## 关键结论

- 创客中心是用户端分销入口，主要服务分销员查看团队、收益和提现，不承载后台管理内容。
- 平台内人人都是分销员，家长和老师均具备分销身份；但后台可以控制具体用户是否开通分销权限。未开通用户在个人中心仍需要有明确创客中心入口，并提示申请开通。
- 分销为二级分销，分销员可看到自己的一级分销员以及每个一级分销员对应的全部二级分销员。
- 当前贡献来源先覆盖注册、买课等；后续对接排课系统后，再展示上课等履约贡献。
- 创客中心主路径调整为：个人中心 → 创客中心入口展示 → 团队成员列表 → 提现中心 → 提现申请。
- 前端用户提交提现申请后，后台收到申请；财务根据申请在线下/系统外执行打款，打款成功后再回到系统内操作“打款成功”完成状态更新。
- 成员贡献金额按系统配置的分成比例计算并展示，用于说明该成员为当前用户带来的分销金额。
- 解释性内容可以与原型同屏，但必须放入说明标签、黄色说明区或画布便签，不得混入真实页面标题、列表、按钮或业务状态。
- 提现时效不做承诺，用户端不能写成最终到账承诺；收款方式仍需确认；金额展示按系统配置分成比例表达，不写死比例或默认金额。

## 1. 背景与范围

本次需求面向平台内所有用户，包括家长和老师。所有用户具备分销身份口径，但是否能看到和使用创客中心由后台分销权限控制。创客中心需要解决“团队看不清、收益来源不透明、提现状态不明确”的问题，并为后续用户端原型提供页面和字段依据。本次包含个人中心入口展示、创客中心首页、团队成员列表、提现中心、提现申请及同屏说明标签规则；不包含后台分销商管理、佣金规则配置、提现审核后台、分销报表、自动打款、多级分销和接口设计。

## 5. PC/移动端差异

移动端：
- 本次创客中心需求主要面向移动端小程序，从个人中心进入。
- 移动端需要覆盖个人中心入口展示、创客中心首页、团队成员列表、提现中心、提现申请和提交/处理中状态。
- 后台分销权限控制不在移动端配置，但移动端需要体现权限结果：已开通用户可进入，未开通用户仍展示入口并提示申请开通。
- 页面空间有限，说明文案不能混入手机页面主体；原型中应放到手机画面外的说明标签或画布便签。

PC 端：
- 用户端创客中心 PC 端本次不作为主范围。
- 后台分销管理、提现审核、佣金规则配置属于后台 PC 场景，但不属于“创客中心用户端”页面。
- 后续如做 PC 管理端，应单独分析后台页面，不应塞进创客中心首屏。

端差异待确认：
- 家长端和教师端都应具备创客中心入口；具体入口位置和端侧文案是否完全一致仍需交互阶段确认。

## 6. 页面与跳转逻辑

本次创客中心原型建议包含以下用户端页面：

1. 个人中心入口展示
- 用于展示创客中心入口和当前用户是否可进入创客中心。
- 从个人中心进入或在个人中心中展示入口。
- 已开通分销权限时，点击入口进入创客中心首页。
- 未开通分销权限时，展示申请开通提示，不进入团队和收益页面。

2. 创客中心首页
- 用于展示团队规模、今日新增、可提现、待结算和团队成员入口。
- 从个人中心入口展示进入。
- 点击团队成员入口进入团队成员列表。

3. 团队成员列表
- 用于展示一级分销员、一级分销员对应的全部二级分销员、成员来源/行为记录和成员贡献金额。
- 从创客中心首页进入。
- 点击成员行可列表内展开或进入成员详情，具体表现待交互确认。
- 点击提现入口进入提现中心。

4. 提现中心
- 用于展示可提现余额、收益记录、提现记录和申请入口。
- 从团队成员列表或创客中心首页提现入口进入。
- 点击“申请提现”进入提现申请。

5. 提现申请
- 用于填写提现金额和收款信息。
- 从提现中心进入。
- 提交成功后展示已提交/处理中状态，并可返回提现中心查看记录。
- 后台收到申请后，财务系统外打款；打款成功后在系统内操作“打款成功”，用户端提现记录状态随之更新。

6. 原型说明标签区
- 不是业务页面，属于原型评审辅助区域。
- 与上述页面同屏展示，用于说明页面目的、入口/出口、字段规则、待确认项。
- 不参与用户真实操作链路。

- 二级分销员在团队成员列表中是以一级分销员下钻、树形分组，还是筛选标签方式展示？
- 提现申请的收款信息采用银行卡、微信零钱、支付宝还是其他方式？
- 可提现金额为 0 时，申请提现按钮是隐藏、置灰还是允许进入后提示？
- 提现记录点击后是否需要详情页/弹窗，还是仅列表展示状态？

### 重点说明

需要用说明标签展示：
- 页面入口/出口。
- 人人都是分销员，家长和老师都能拉新并获得分润。
- 后台可控制用户是否开通分销权限；未开通用户仍展示入口，并提示申请开通。
- 二级分销定义：当前用户可查看一级分销员，以及每个一级分销员对应的全部二级分销员。
- 当前展示注册、买课等分润触发范围；上课贡献待后续排课系统对接后展示。
- 成员贡献金额按系统配置的分成比例展示，不在用户端写死比例。
- 可提现、待结算、审核中等状态说明。
- 提现申请由用户提交，后台收到后由财务系统外打款，打款成功后系统内标记成功；提现时效不做承诺。
- 收款信息方式待确认。

不应放入产品 UI 主体：
- 后台管理入口。
- 流程图入口。
- 原型评审按钮。
- 大段说明文字。
- 未确认规则的承诺性文案。

### 待确认但可先占位

- 申请开通分销权限的承接方式。
- 二级分销员列表展示形式。
- 收款方式。

## 风险与依赖

- 如果把后台管理内容放进创客中心默认首屏，会再次偏离已确认纠偏结论。
- 如果把解释性文案伪装成产品字段或按钮，会导致评审误判为真实 UI。
- 金额和收款方式未确认前，原型只能用说明标签标注待确认，不能形成用户承诺；提现时效已明确不做承诺。
- 家长端和教师端入口位置、展示文案未确认，会影响后续信息架构和原型文案。
