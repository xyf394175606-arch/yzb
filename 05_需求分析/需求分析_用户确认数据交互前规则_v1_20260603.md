# 用户确认数据交互前规则

## 背景

本文件记录用户在进入 `06 数据交互` 前补充确认的业务规则，用于覆盖上游产物中仍标记为待确认的分销、提现、领榜接单金额和无人抢单处理规则。

## 目标

- 固化 06 数据交互前必须明确的业务规则。
- 为 `09_数据交互/` 的页面数据契约、字段结构、状态规则和 mock 数据提供依据。
- 避免后续助手继续把已确认事项作为待确认问题处理。

## 关键结论

- 分销比例、注册奖励、课消提成具体数值不写死，后台按可配置比例或金额处理，PRD 标记为运营配置项。
- 分销佣金触发条件已确认：支付成功后生成候选佣金；订单完成或课消完成后转为可结算；退款或退单时佣金失效或冲正。
- 提现审核角色已确认：财务审核，管理员可查看和复核，客服无提现审核权限。
- 领榜老师付费购买需求单的金额已确认：按需求单金额的 3 倍课时费计算。
- 领榜无人抢单时不做系统派单，客服可以人工发送推送消息给老师，提醒老师该需求单可抢。
- 领榜老师抢单资格已确认：老师需在平台提交认证资料；认证成功后可正常接单。未认证老师允许查看未被老师接单的需求单，也允许付款接单，但前端必须提醒其进行认证。
- 退款人工处理责任已确认：退款属于系统外人工处理事项，当前系统不处理退款责任和财务归属。
- 分销归因有效期已确认：拉新注册归因永久有效；销售分销不设置有效期，购买时通过哪个分享链接成交即归属该分享人；若购买时没有销售分享人，则销售分销提成归属于注册归因人。
- 客服推送老师规则已确认：推送渠道包含微信服务号、企业微信机器人、短信通知；频率暂按 3 天一次；推送范围为符合需求单要求的老师，例如性别、科目、年级等。
- 第三方排课后续对接方式已确认：通过 API 接口形式对接；当前仍保留最小排课闭环，后续在数据交互中预留 API 同步、失败和冲突状态。
- 优坐标会员卡权益已确认：会员卡只属于优坐标范围。会员卡的作用是获得会员购课权益，例如课程对外售价 199，会员价 129。会员分普通会员和卓越会员；普通会员有优惠购课次数限制；卓越会员有效期 1 年，不限制购课数量，但一次给 100 节课时购买机会，用完后可联系客服补充。

## 已确认规则

| 规则项 | 已确认处理方式 | 影响阶段 |
| --- | --- | --- |
| 分销比例 | 不写死数值，后台配置一级/二级销售提成比例 | 数据交互、PRD、后台规则配置 |
| 注册奖励 | 不写死数值，后台配置奖励金额或比例 | 数据交互、PRD、后台规则配置 |
| 课消提成 | 不写死数值，后台配置课消提成比例；课消提成仅一级 | 数据交互、PRD、佣金结算 |
| 佣金候选生成 | 支付成功后生成候选佣金 | 数据交互、佣金状态 |
| 佣金可结算 | 订单完成或课消完成后转为可结算 | 数据交互、佣金状态、提现 |
| 佣金失效或冲正 | 退款或退单时佣金失效或冲正 | 数据交互、异常状态、财务记录 |
| 提现审核角色 | 财务审核，管理员可查看和复核，客服无审核权限 | 权限、提现审核、后台页面 |
| 领榜接单金额 | 老师付费购买金额为需求单金额的 3 倍课时费 | 领榜老师端、支付、财务展示 |
| 领榜无人抢单处理 | 客服可人工发送推送消息给老师，提醒需求单可抢 | 领榜客服后台、异常处理、消息推送 |
| 老师抢单资格 | 认证成功后可正常接单；未认证可看未接单需求单并可付款接单，但前端提醒认证 | 老师端、抢单、认证提示 |
| 退款人工处理 | 退款责任和财务归属属于系统外事项，当前系统不处理 | 异常处理、PRD 边界 |
| 分销归因有效期 | 拉新注册永久有效；销售按购买时分享链接归因；无销售分享人时归属于注册归因人 | 分销、佣金、归因 |
| 客服推送老师 | 微信服务号、企业微信机器人、短信通知；3 天一次；范围为符合需求单要求的老师 | 推送记录、通知状态 |
| 第三方排课对接 | 后续通过 API 接口对接；当前保留最小排课闭环并预留同步状态 | 优坐标排课、数据交互 |
| 优坐标会员卡权益 | 仅属于优坐标；普通会员限次优惠购课；卓越会员 1 年有效，不限购课数量，一次给 100 节课时购买机会，用完联系客服补充 | 优坐标会员、产品购买、订单、权益 |

## 对 06 数据交互的要求

- 分销规则需要设计为配置型数据，不在接口字段中写死固定数值。
- 佣金状态至少覆盖：候选、可结算、可提现、提现审核中、待打款、已到账、已驳回、已失效、已冲正。
- 佣金触发需区分销售提成、注册奖励和课消提成。
- 提现审核接口场景仅允许财务执行审核、驳回、确认打款；管理员只读和复核；客服不可访问审核动作。
- 领榜接单金额需支持按需求单课时费计算 3 倍金额，并在老师端接单确认、支付结果、后台抢单记录中展示。
- 领榜无人抢单需要支持客服触发人工推送提醒老师，记录推送时间、推送对象、操作客服和推送结果。
- 老师抢单上下文需要返回认证状态、是否已认证、认证提醒文案；未认证但付款接单时不阻断接单，只展示认证提醒。
- 推送提醒数据需要支持渠道、频率、目标老师筛选条件、目标老师列表、发送结果、失败原因和下次可推送时间。
- 分销归因数据需要区分注册归因人和销售分享人；购买时有分享人则销售提成归属分享人，无分享人则归属注册归因人。
- 第三方排课 API 对接需预留同步状态、同步失败原因、重试状态和冲突处理字段，但当前原型仍按最小排课闭环展示。
- 优坐标会员卡需要支持会员等级、对外售价、会员价、优惠购课次数、卓越会员有效期、课时购买机会剩余数、联系客服补充入口等数据；会员权益不得应用到领榜。

## 风险与依赖

- 若后续要求固定佣金比例，需要回填运营配置默认值，但不影响当前数据契约建模。
- 若后续要求系统自动派单，需要回流业务流程和信息架构；当前只做客服人工推送提醒。
- 若后续引入系统内退款，需要同步调整佣金冲正规则和财务流程；当前退款责任和财务归属不进入系统。
