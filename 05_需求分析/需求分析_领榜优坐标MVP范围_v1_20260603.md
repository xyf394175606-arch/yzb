# 需求分析稿

## 背景

本稿基于 `02_会议分析/` 的会议分析交接稿、批量会议纪要、重复与缺失核对清单，以及用户补充业务边界整理。当前实际项目为 `领榜教育` 和 `优坐标`；两个项目各自有自己的后台；两个项目都面对家长端和老师端。历史会议中存在共用后台、数据隔离、共享资料、分销、会员、排课等多条线索，本稿按时间线和当前边界进行需求成型，不把早期会议观点直接当成最终需求。

## 目标

形成可确认、可排序、可验收、可交给后续设计链路使用的需求基础，明确已确认需求、推测需求、待确认需求、暂不纳入范围、MVP 候选范围、优先级和验收标准。本稿不输出业务流程、页面交互、接口字段、高保真原型或最终 PRD。

## 关键结论

- 已确认的最高层边界是：做 `领榜教育` 与 `优坐标` 两个项目，两个项目各自有后台，且都面对家长端和老师端。
- 领榜教育的当前需求线索更偏家教撮合：家长发布或填写需求、预约老师、老师入驻并响应需求、客服/教务可能介入。
- 优坐标的当前需求线索更偏课程/老师资源展示、会员/课包、批发或 B 端学习交付，并存在排课课销对接线索。
- 最大冲突是“两个项目各自后台”与历史会议中的“共用后台、总后台、数据隔离、家长/老师资料整合”。本稿将“各自后台”作为已确认边界，将“共享中台/共享资料”列为待确认或推测，不写成已确认需求。
- MVP 可先围绕两个项目的基础端侧能力、各自后台基础管理、必要运营配置和明确的客服/教务协作能力候选；分销、提现、会员复杂账务、第三方排课深度对接不宜在未确认前被强行纳入。

| 优坐标家长端：课程/老师/会员或咨询基础入口 | 候选纳入 | 优坐标家长端已确认，会议有展示与会员线索 | 需由业务方确定优先定位 |
| 优坐标老师端：老师资料展示或交付关联能力 | 候选纳入 | 优坐标也面对老师端 | 是否进入 MVP 取决于优坐标主线 |
| 优坐标后台：资源、会员、运营配置基础管理 | 候选纳入 | 后台已确认各自存在，运营配置反复出现 | 会员规则未确认前仅保留基础配置候选 |
| 共用能力：家长/老师资料共享或同步 | 待确认后再定 | 与当前“各自后台”存在冲突 | 不建议未确认直接纳入 |
| 分销/提现/财务审核 | 待确认后再定 | 有明确线索但账务和风控风险高 | 可作为 P2 或后续版本候选 |
| 第三方排课对接 | 待确认后再定 | 有明确线索但依赖延期 | 若业务强依赖，再列 MVP 风险项 |
| 操作手册和推广测试二维码 | 候选纳入 | 会议待办明确，利于运营测试 | 可作为上线准备项，不扩大功能范围 |

## 优先级

| 需求 | 优先级 | 判断依据 | 依赖 |
| --- | --- | --- | --- |
| 明确两个项目、两个后台、双端拆分边界 | P0 | 用户补充明确，影响全部下游 | 无 |
| 澄清共享中台/共享资料边界 | P0 | 与后台边界冲突，影响权限和数据 | 业务负责人、技术负责人确认 |
| 领榜教育家长需求单与老师响应机制 | P0 | 领榜核心闭环线索 | 业务负责人确认 |
| 领榜教育后台基础管理候选 | P0 | 领榜闭环需要后台承接 | 领榜业务机制确认 |
| 优坐标当前主定位确认 | P0 | 决定会员、课程展示、B 端交付和排课优先级 | 业务负责人确认 |
| 优坐标基础资源/会员配置候选 | P1 | 后期会议多次出现会员和资源线索 | 优坐标定位确认 |
| 客服/教务/运营角色边界 | P1 | 影响后台权限和协作 | 岗位负责人确认 |
| 排课系统对接 | P1 | 影响优坐标交付，但外部依赖高 | 技术负责人、第三方接口 |
| 分销/提现/佣金 | P2 | 会议有线索但财务复杂度高 | 财务规则、风控规则确认 |
| AI/英语工具/企业微信监控 | P3 | 非基础闭环前置 | 后续版本规划 |

## 验收标准

| 需求 | 验收标准 | 不包含内容 |
| --- | --- | --- |
| 两项目拆分边界 | 需求文档和后续流程能分别呈现领榜教育、优坐标，不混用项目名称 | 不包含具体页面结构或接口字段 |
| 两后台默认独立 | 后续分析中默认存在领榜后台和优坐标后台；共享数据能力单独标记为待确认 | 不默认存在总后台或共享中台 |
| 双端拆分 | 每个项目均能列出家长端、老师端、后台的需求候选和待确认项 | 不要求两个项目端侧功能完全一致 |
| 领榜家长需求候选 | 能描述家长提交家教需求、预约或请求老师响应的业务目标、用户价值和待确认机制 | 不定义具体表单字段、页面步骤或接口 |
| 领榜老师响应候选 | 能描述老师入驻/展示/响应需求的价值和不确定点 | 不定义抢单、派单、接单的最终流程 |
| 优坐标基础能力候选 | 能区分课程/老师展示、会员/课包、B 端交付、排课对接等不同线索 | 不替业务方确定优坐标最终定位 |
| 后台基础管理候选 | 能列出家长资料、老师资料、运营配置、权限边界和项目归属待确认点 | 不定义后台菜单、操作路径或数据字段 |
| 分销/提现候选 | 能说明其业务价值、风险和待确认账务规则 | 不承诺进入 MVP，不输出财务流程 |
| 排课对接候选 | 能说明第三方依赖、延期风险和是否需确认 MVP 归属 | 不输出接口方案或降级流程 |

## 下游交接说明

| 下游助手 | 需要关注的内容 |
| --- | --- |
| 业务流程助手 | 可以开始，但必须以“候选流程 + 待确认点”方式处理。优先拆领榜教育、优坐标、各自后台、家长端、老师端和后台角色；重点处理两个后台 vs 共享能力冲突。 |
| 信息架构助手 | 等业务流程初稿后，再按两个项目、双端、各自后台和可能的共用能力拆模块，不要提前假设总后台。 |
| 交互设计助手 | 等业务流程与信息架构明确后，再处理端侧操作路径；不要把需求分析中的候选直接画成最终交互。 |
| 数据交互助手 | 必须等待后台边界、共享资料规则、分销/提现和排课归属确认后再定义接口字段。 |
| 高保真原型助手 | 必须等待流程、信息架构、交互和数据交互产物，不应基于当前稿直接做最终原型。 |

## 风险与依赖

- 当前虽然可以派发业务流程助手，但业务流程助手必须保留关键待确认项，不能把候选 MVP 当成最终范围。
- 最大风险是后台边界：用户确认各自后台，但会议材料有大量共用后台、数据隔离、共享资料线索。
- 领榜教育的老师响应机制和优坐标的商业定位未确认，会影响主流程和优先级。
- 分销/提现/佣金涉及财务和微信风控，排课涉及第三方接口，均不宜直接作为无风险 MVP。
- 公开竞品/同行资料本次未触发；后续如需对标，只能作为参考和风险提示。
- 空文件未补充前不可引用，疑似同源材料不应重复计数。
