# 需求分析：优坐标渠道商名额闭环

## 1. 背景与范围

本稿单独分析优坐标“渠道商名额闭环”，覆盖渠道商创建、名额充值、给家长开通/续费会员、名额消耗/转让和记录追踪。输入来源为 Axure 原型中的 `渠道商管理`、`渠道商端`、`会员管理`、`会员充值订单`，以及 v1/v2 需求分析稿。

本闭环目标是把渠道商后台管理和渠道商端自助操作拆清楚，同时标明其与会员权益闭环的交叉关系。本稿不输出 API、数据库或实现方案。

关键结论：

- 已确认需求：后台可新增/编辑渠道商、充值会员名额、查看消耗记录和充值记录；渠道商端可给家长开通/续费会员、转让名额、导出权益包列表。
- 推测需求：申请合伙人线索可转为渠道商，但当前有效页面是否沿用历史版本需确认。
- 待确认需求：渠道商开通的会员类型、验证码接收账号、关闭渠道商后的权益和名额处理。
- 暂不纳入范围：未确认的合伙人管理占位页和历史申请列表主链路。

## 2. 业务规则

- 渠道商可由后台直接新增，也可由合伙人/招商线索转化生成；联系电话在新增后作为唯一登录账号，编辑时不可编辑。
- 渠道商状态为关闭时，该账号不能登录 PC 后台和小程序端。
- 后台可为渠道商充值会员名额，充值后增加购买会员名额数量和剩余名额数量，并生成充值记录。
- 渠道商端可给家长开通/续费会员；每次开通/续费消耗 1 个会员名额，并为家长开通会员身份及充值 100 折扣课时。
- 渠道商端可转让名额；确认后扣减当前渠道商剩余可分配额度，增加接收方渠道商额度，并生成转让记录。
- 渠道商端权益包列表应与后台 `会员管理` 中的会员权益包同步。
- 一个后台页面可支撑多个闭环：`会员管理` 同时服务会员权益闭环和渠道商名额闭环。

## 6. PC/移动端差异

PC 端：

- `渠道商管理` 是平台后台管理页面，负责主档、充值和记录查询。
- `渠道商端` 具有独立角色端属性，可视为渠道商登录后的操作端，形式上来自后台页面组但业务上属于渠道商侧。

移动端：

- 当前小程序用户侧主要体现会员权益结果，渠道商端是否有移动端入口未在原型中明确。

历史/暂废：

- `申请合伙人列表（2026_6_5）` 可作为渠道商来源参考，不进入当前主链路，除非确认沿用。

## 8. 页面与跳转逻辑

```text
后台渠道商管理新增/编辑渠道商
-> 联系电话成为渠道商登录账号
-> 后台为渠道商充值会员名额
-> 渠道商端展示总购入额度和剩余可分配
-> 渠道商端给家长开通/续费会员
-> 扣减 1 个名额并生成消耗记录
-> 会员管理同步会员权益包
-> 我的会员权益展示权益结果
```

名额转让支线：

```text
渠道商端发起转让名额
-> 输入接收方和数量
-> 确认后扣减当前渠道商剩余名额
-> 增加接收方剩余名额
-> 生成转让记录
```

## 9. 待确认问题

- 渠道商端开通/续费会员时，验证码发送给家长账号还是渠道商账号。
- 渠道商关闭后，已开通会员、剩余名额、充值记录、消耗记录是否继续可见或生效。
- 渠道商开通会员时是否可选择普通会员/卓越会员，或固定开通某一权益包。
- 接收方渠道商不存在、关闭或名额转让数量超过余额时如何提示。
- 申请合伙人列表是否继续作为渠道商来源页面。

## 10. 风险与依赖

- 渠道商名额闭环依赖会员权益闭环生成权益包。
- 名额充值、转让和开通涉及权益资产，需业务和财务确认规则。
- 渠道商账号关闭规则若不明确，会影响历史权益、登录和数据追踪。

## 11. 下一步动作

- 业务流程阶段拆出渠道商创建、充值、开通会员、续费、转让、关闭六条流程。
- 信息架构阶段确认 `渠道商端` 是后台角色页还是独立渠道商端入口。
- 交互设计阶段补齐名额不足、账号不存在、验证码错误、关闭账号等异常。

## 12. 原型设计输入

- 页面范围：渠道商管理、渠道商端、会员管理、会员充值订单、申请合伙人列表参考。
- 字段批注：联系电话作为账号、购买会员名额、剩余名额、消耗记录、充值记录、会员权益包同步。
- 按钮批注：新增、编辑、充值、开通会员、续费、转让名额、导出。
- 黄色说明区：名额充值、扣减、转让、关闭状态、验证码和会员类型待确认。
