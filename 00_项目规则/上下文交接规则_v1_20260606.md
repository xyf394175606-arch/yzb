# 上下文交接规则

## 背景

k12 项目采用多对话、多助手协作方式推进。主控助手和各阶段助手都可能因为聊天上下文过长而无法继续完整承接任务，因此必须建立统一的上下文交接规则，确保项目推进不依赖聊天记忆。

## 目标

- 让主控助手和阶段助手在上下文接近满或已经满时可被新助手接替。
- 通过项目文件完成交接，不依赖旧对话记忆。
- 统一接替助手命名方式，避免多个接替线程混淆。
- 明确旧助手、新助手和项目主控的责任边界。

## 输入来源

- `AGENTS.md`
- `00_项目规则/项目总览_工作流程_v1_20260602.md`
- `00_项目规则/项目主控调度规则_v1_20260602.md`
- `00_项目规则/助手向主控回报规则_v1_20260603.md`
- `00_项目规则/阶段自动交接规则_v1_20260603.md`
- `13_任务卡片/`

## 关键结论

任何助手发生上下文交接时，必须优先通过交接文件完成项目状态传递。新助手不得只依赖聊天摘要继续工作。

新助手命名必须采用：

```text
原有助手名称 V版本号 交接日期
```

其中：

- `原有助手名称` 使用原线程标题或项目内标准助手名称，例如 `07 - 高保真原型助手`。
- `V版本号` 从 `V0.1` 开始；同一原助手再次交接时依次递增为 `V0.2`、`V0.3`。
- `交接日期` 精确到日，格式为 `YYYYMMDD`，例如 `20260606`。

示例：

- `大统领 V0.1 20260606`
- `07 - 高保真原型助手 V0.1 20260606`
- `09 - 评审助手 V0.2 20260608`

## 适用范围

| 类型 | 适用对象 | 交接文件位置 |
| --- | --- | --- |
| 主控交接 | 大统领 / 项目主控助手 | `outputs/` |
| 阶段助手交接 | 01-09 号业务阶段助手 | 对应阶段目录 |
| 备份助手交接 | 10 号备份助手 | `outputs/` 或备份规则指定位置 |
| 临时接替交接 | 旧助手已无法输出交接文件时，由新助手补写 | 按接替对象类型确定 |

## 交接触发条件

以下任一情况出现时，必须执行上下文交接：

- 助手明确发现当前对话上下文接近上限。
- 助手回复质量开始受上下文长度影响，无法稳定读取完整上游信息。
- 用户要求新助手接替旧助手。
- 旧助手线程系统错误、中断、不可读取或无法继续派发。
- 主控判断某助手已无法安全继续承担当前阶段任务。

## 交接文件命名

交接文件仍必须遵守项目文件命名规则：

```text
文件主题_调整内容_版本号_创建时间
```

推荐命名：

| 场景 | 文件名示例 |
| --- | --- |
| 主控主动交接 | `上下文交接_主控交接_v1_20260606.md` |
| 主控接替补写 | `上下文交接_主控接替_v1_20260606.md` |
| 阶段助手主动交接 | `上下文交接_高保真原型助手交接_v1_20260606.md` |
| 阶段助手接替补写 | `上下文交接_评审助手接替_v1_20260606.md` |

说明：线程命名规则和文件命名规则不同。线程标题按 `原有助手名称 V版本号 交接日期`，交接文件按项目文件命名规则。

## 主控交接要求

主控助手交接文件必须包含：

- 背景
- 目标
- 输入来源
- 当前项目阶段
- 当前主控判断
- 已完成阶段和最新有效产物
- 已派发但未完成的助手任务
- 阻塞问题
- 非阻塞待确认问题
- 风险与依赖
- 下一步调度建议
- 旧线程名称和线程 ID
- 新线程名称和线程 ID
- 必须继续遵守的规则文件

主控新线程启动后，必须先读取：

- `AGENTS.md`
- `00_项目规则/项目总览_工作流程_v1_20260602.md`
- `00_项目规则/项目主控状态台账_v1_20260602.md`
- `00_项目规则/上下文交接规则_v1_20260606.md`
- 最新主控交接文件
- 当前阶段所需上游产物

## 阶段助手交接要求

阶段助手交接文件必须包含：

- 背景
- 目标
- 输入来源
- 本助手负责范围
- 本次任务目标
- 已读取的输入文件
- 已完成判断
- 已生成或修改的文件
- 尚未完成的工作
- 当前不能改的业务结论
- 待确认问题
- 风险与依赖
- 下游交接包
- 接替助手启动提示

阶段助手新线程启动后，必须先读取：

- `AGENTS.md`
- `00_项目规则/项目总览_工作流程_v1_20260602.md`
- `00_项目规则/助手向主控回报规则_v1_20260603.md`
- `00_项目规则/上下文交接规则_v1_20260606.md`
- 自己对应的 `13_任务卡片/` 任务说明
- 最新阶段交接文件
- 当前阶段所需上游产物

## 接替助手命名规则

当某个助手发生交接并创建新助手线程时，主控必须按以下规则命名新线程：

```text
原有助手名称 V版本号 交接日期
```

版本号规则：

- 第一次接替使用 `V0.1`。
- 第二次接替使用 `V0.2`。
- 第三次接替使用 `V0.3`。
- 版本号只表示上下文接替次数，不表示阶段产物版本。

日期规则：

- 交接日期使用新助手创建或正式接管当天日期。
- 日期格式为 `YYYYMMDD`。
- 不记录小时分钟，避免线程标题过长。

示例：

| 原助手 | 新助手线程标题 |
| --- | --- |
| `大统领` | `大统领 V0.1 20260606` |
| `02 - 需求分析助手` | `02 - 需求分析助手 V0.1 20260606` |
| `07 - 高保真原型助手 V0.1 20260606` | `07 - 高保真原型助手 V0.2 20260608` |

## 主控处理要求

主控发现或收到助手交接需求后，必须：

1. 判断旧助手是否还能输出交接文件。
2. 如果旧助手能继续响应，先要求旧助手输出交接摘要或交接文件。
3. 如果旧助手已无法响应，由主控或新助手基于项目文件补写接替交接文件。
4. 创建或启用新助手线程，并按命名规则命名。
5. 要求新助手读取交接文件、任务卡片、上游产物和通用规则。
6. 在主控状态台账中记录接替关系。
7. 向用户汇报新旧助手交接结果和下一步动作。

## 禁止事项

- 不得让新助手只依赖旧聊天摘要继续工作。
- 不得在没有交接文件的情况下长期推进。
- 不得把交接文件当成正式阶段产物替代需求、流程、交互、数据、原型、PRD 或评审记录。
- 不得用阶段产物版本号代替助手交接版本号。
- 不得创建多个同名接替助手而不标注版本号和交接日期。
- 不得让阶段助手绕过主控自行交接给新线程。

## 待确认问题

- 当前无新增待确认问题。本规则作为主控和 01-10 号助手的统一上下文交接规则执行。

## 风险与依赖

- 如果交接文件不完整，新助手可能误用旧结论、旧路径或过期原型版本。
- 如果新助手命名不带版本号和日期，后续主控无法判断哪个线程是最新接替者。
- 如果旧助手已满且没有主动交接，新助手必须从项目文件反推状态，可能需要更多检查时间。

## 下一步动作

- 将本规则加入主控必读文件。
- 将本规则加入助手回报规则。
- 后续主控派发任务时，如涉及接替线程，必须明确新助手线程命名和交接文件路径。
