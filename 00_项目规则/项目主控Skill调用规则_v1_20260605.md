# 项目主控 Skill 调用规则

## 背景

k12 项目主控助手可根据任务需要识别并使用 Figma、文档、表格、浏览器、图像、PDF、PPT 等 skill 或插件能力。但 skill 调用可能影响产物方向、外部资料引用和阶段助手输入，因此必须建立调用前确认和结果沉淀规则。

## 目标

确保主控能主动识别适合的 skill，同时避免未经用户确认就调用外部能力或把外部方法覆盖到 k12 已确认业务规则上。

## 关键结论

主控可以判断应使用哪个 skill，但实际调用 skill 前必须先询问用户。用户确认后，主控再调用对应 skill，并把结果沉淀进项目文件、任务包或阶段交接材料。

skill 结果只能作为输入、参考或验证材料，不得覆盖 k12 项目文件和用户已确认业务规则。

## Skill 识别规则

| 输入或任务 | 可能适用 skill | 用途 |
| --- | --- | --- |
| Figma 链接、设计稿、组件库 | Figma skill | 读取设计上下文、截图、变量、组件结构 |
| 本地 HTML、网页、原型验证 | Browser 或 Chrome skill | 打开、截图、检查交互和页面状态 |
| 图片、截图、视觉参考 | 图像或视觉检查能力 | 识别页面结构、视觉差异、内容信息 |
| PDF、合同、资料包 | PDF skill | 提取内容、检查页码和结构 |
| Word 文档、PRD 文档 | Documents skill | 创建、编辑、检查文档交付物 |
| 表格、佣金、数据清单 | Spreadsheets skill | 整理结构化数据、计算、校验 |
| PPT 或汇报材料 | Presentations skill | 生成或检查演示材料 |
| GitHub 方法库或外部资料 | 浏览器或网络检索能力 | 读取方法参考，但不得覆盖项目规则 |

## 调用前确认规则

主控发现任务适合调用 skill 时，必须先向用户说明：

- 建议调用哪个 skill。
- 调用目的是什么。
- 会读取或处理哪些资料。
- 调用结果会交给哪个助手或写入哪个项目文件。
- 是否会影响当前阶段结论。

用户确认前，主控不得调用该 skill。

## 调用后沉淀规则

skill 调用完成后，主控必须把结果沉淀到以下至少一种位置：

- 助手任务包。
- `work/` 下的材料识别、调研记录、差异清单或交接清单。
- 对应阶段目录的正式产物。
- 主控状态台账或当前状态简报。

不得只在聊天中保留 skill 调用结论。

## 外部资料优先级

| 优先级 | 来源 |
| --- | --- |
| 最高 | 用户明确确认的业务规则 |
| 高 | k12 项目正式阶段产物 |
| 中 | 主控翻译稿和任务交接包 |
| 参考 | skill 输出、外部资料、第三方方法库 |

如果 skill 输出或外部方法与 k12 已确认规则冲突，必须以 k12 已确认规则为准，并把冲突记录为问题。

## 禁止事项

- 不得在用户确认前调用 skill。
- 不得把 skill 输出直接写成已确认需求。
- 不得让外部方法库覆盖 k12 已确认业务规则。
- 不得把 skill 调用结果只留在聊天里。
- 不得把需要用户确认的外部资料判断交给阶段助手自行决定。

## 待确认问题

当前默认执行口径已确认：skill 调用前先问用户；确认后再调用；调用结果必须沉淀到项目文件或任务包。

## 风险与依赖

未经确认调用 skill 可能造成外部资料误用、设计方向偏移或阶段助手输入不一致。

## 下一步动作

- 主控派发任务时，如涉及 skill，应先确认调用。
- 确认后主控调用 skill，并把结果写入任务包或项目文件。
- 阶段助手使用 skill 结果时，必须以 k12 项目已确认规则为最高优先级。
