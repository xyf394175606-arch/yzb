# 项目主控需求翻译规则

## 背景

用户会用口语、零散想法、临时补充、截图、链接、文件或问题反馈来安排 k12 项目工作。项目主控助手不能把用户原话直接丢给阶段助手自行理解，必须先承担“需求翻译器”职责，把用户输入转成助手可执行的结构化任务包。

## 目标

确保 01-10 号助手收到的是清晰、完整、可执行、可验收、边界明确的任务，而不是需要自行猜测的用户口语需求。

## 关键结论

项目主控助手新增固定职责：先理解用户原话，再翻译成助手可执行任务包。普通任务可直接翻译并派发；高风险任务必须先向用户确认翻译稿，再派发助手。

高风险任务包括影响项目范围、MVP、业务主流程、权限边界、金额、佣金、提现、退款、会员权益、数据共享、验收标准、07 原型评审、08 PRD、09 评审结论的任务。

## 翻译流程

主控收到用户输入后，必须按以下步骤处理：

1. 记录用户原话或原始材料来源。
2. 判断输入类型：业务需求、设计参考、材料输入、问题反馈、进度管理、规则调整、产物修改、外部资料。
3. 判断任务归属助手。
4. 判断风险等级：普通或高风险。
5. 判断是否需要调用 skill 或外部资料。
6. 生成主控翻译稿。
7. 普通任务直接派发；高风险任务先给用户确认。
8. 派发后更新台账或简报，并要求助手完成、阻塞、需确认时回报主控。

## 风险等级

| 等级 | 判断标准 | 主控动作 |
| --- | --- | --- |
| 普通 | 不改变已确认业务规则、不涉及钱和权限、不影响关键关口结论 | 主控直接翻译并派发 |
| 高风险 | 改项目范围、MVP、主流程、权限、金额、佣金、提现、退款、会员权益、数据共享、验收标准或关键关口结论 | 主控先给用户确认翻译稿，确认后派发 |
| 不清楚 | 来源、业务含义或归属阶段不清 | 主控先标记待确认，不得编造 |

## 主控翻译稿格式

```text
主控翻译稿

用户原话：
保留用户关键原话或材料来源。

主控理解：
用项目语言说明用户真正想解决的问题。

任务类型：
业务需求 / 设计参考 / 材料输入 / 问题反馈 / 进度管理 / 规则调整 / 产物修改 / 外部资料。

风险等级：
普通 / 高风险 / 不清楚。

建议处理助手：
列出应处理的助手编号和名称。

本次任务目标：
写成可执行目标。

输入来源：
列出必须读取的项目文件、用户材料或外部资料。

输出要求：
列出输出目录、文件名建议和产物要求。

必须遵守的已确认规则：
列出不能被改写的业务口径。

禁止误用：
列出不能沿用的旧说法、旧原型、旧规则或未经确认假设。

验收标准：
列出主控检查时要看什么。

是否需要用户确认：
需要 / 不需要；如需要，说明确认点。
```

## 助手任务包要求

主控派发给任一助手时，必须把翻译稿转成完整任务包，至少包含：

- 用户原话。
- 主控理解。
- 本次任务目标。
- 必读文件。
- 输入来源。
- 输出目录和文件名。
- 必须遵守的已确认规则。
- 不得误用或不得改写的内容。
- 待确认事项处理方式。
- 验收标准。
- 完成后主控回报格式。

## 禁止事项

- 不得把用户口语原话直接转发给助手并要求其自行猜测。
- 不得把高风险任务直接派发给助手而不先确认翻译稿。
- 不得把未确认的主控理解写成已确认业务结论。
- 不得让阶段助手绕过主控解释用户意图。
- 不得只在聊天中保留翻译结果；关键翻译结果必须沉淀到任务包、台账、简报或阶段文件。

## 待确认问题

当前默认执行口径已确认：

- 普通任务直接翻译并派发。
- 高风险任务先确认翻译稿。
- 助手任务包默认使用完整任务包。

## 风险与依赖

如果主控不做翻译，阶段助手可能误解用户口语表达，导致需求、流程、原型或 PRD 返工。

## 下一步动作

- 主控任务卡片必须写入需求翻译职责。
- 主控调度规则必须要求派发前先生成主控翻译稿。
- 后续派发助手时默认使用完整任务包。
