# 阶段自动交接规则

## 背景

k12 项目采用 01-09 号业务阶段助手和 10 号备份助手协作。当前已建立“完成/阻塞/需确认时回报主控”的规则，但还需要明确：某一阶段完成且没有问题时，主控应自动把产物交接给下一阶段助手，而不是等待用户反复询问或手动要求继续。

## 目标

- 建立阶段完成后的自动交接机制。
- 明确哪些阶段可以自动交接，哪些关键关口必须先等用户确认。
- 规定阶段助手完成时必须提供下游交接信息。
- 规定主控检查、备份、汇报和派发下一助手的顺序。

## 关键结论

阶段完成后，若产物检查通过、无阻塞、无需要用户拍板的问题，且不属于关键确认关口，主控必须自动交接给下一阶段助手。

自动交接不是静默推进。主控仍必须检查产物、更新台账、触发备份，并向用户汇报“已完成什么、已交给谁、下一步是什么”。

## 自动交接链路

| 完成阶段 | 下游助手 | 是否可自动交接 | 说明 |
| --- | --- | --- | --- |
| 01 会议整理 | 02 需求分析助手 | 是 | 会议纪要和需求分析交接稿检查通过后自动交接 |
| 02 需求分析 | 03 业务流程助手 | 否 | 需求范围、MVP、待确认需求属于关键关口，必须用户确认 |
| 03 业务流程 | 04 信息架构助手 | 否 | 主流程、异常流程、权限边界属于关键关口，必须用户确认 |
| 04 信息架构 | 05 UI/UX 设计助手 | 是 | 页面结构和导航关系检查通过后自动交接 |
| 05 UI/UX 设计 | 06 数据交互助手 | 是 | UI/UX 方案检查通过且无阻塞确认项后自动交接 |
| 06 数据交互 | 07 高保真原型助手 | 是 | 页面数据契约和 mock 数据检查通过后自动交接 |
| 07 高保真原型 | 08 需求文档助手 | 条件自动 | 仅在原型完成且无评审问题、无回流项时可自动交接；原型评审后必须确认 |
| 08 需求文档 | 09 评审助手 | 否 | 最终 PRD 完成后必须用户确认是否进入全链路评审 |
| 09 评审 | 结束或回流 | 否 | 由评审结论决定通过、带问题通过或阻塞回流 |

## 自动交接条件

主控只有在同时满足以下条件时才可自动交接：

- 上游助手已按“主控回报”格式回报完成。
- 正式产物写入正确阶段目录。
- 文件命名符合项目命名规则。
- 产物包含通用输出要求：背景、目标、输入来源、关键结论、待确认问题、风险与依赖、下一步动作。
- 产物未越界替代其他助手职责。
- 无阻塞问题。
- 无必须用户拍板的问题。
- 无需要先回流上游的问题。
- 不属于本规则列出的关键确认关口。

## 自动交接流程

1. 助手完成阶段产物并回报主控。
2. 主控检查产物和回报内容。
3. 主控判断是否属于关键确认关口。
4. 如属于关键确认关口，主控向用户汇报并等待确认，不自动交接。
5. 如不属于关键确认关口且检查通过，主控触发 10 号备份助手执行节点备份。
6. 主控向下游助手派发任务，明确上游输入、输出目录、禁止事项和回报要求。
7. 主控更新 `项目主控状态台账` 和 `项目进度_当前状态简报`。
8. 主控向用户汇报：当前阶段已完成、已自动交接给哪个助手、当前无阻塞确认项。

## 阶段助手交接包要求

阶段助手完成回报时，除统一回报格式外，还必须提供下游交接包：

| 项目 | 要求 |
| --- | --- |
| 建议下游助手 | 明确建议交给哪个助手 |
| 下游输入文件 | 列出下游必须读取的正式产物路径 |
| 已确认结论 | 列出下游可直接采用的结论 |
| 待确认问题 | 区分阻塞与非阻塞 |
| 风险与依赖 | 说明是否影响下游启动 |
| 禁止下游误用 | 明确哪些旧表述、推测或暂不纳入内容不得当作确认规则 |

## 禁止自动交接的情况

- 用户明确要求暂停、先讨论或先确认。
- 出现需求范围、MVP、项目边界不确定。
- 出现主流程、异常流程、状态流转、权限边界不确定。
- 出现原型评审问题，需要判断修复、回流或进入 PRD。
- 最终 PRD 存在未闭环关键问题。
- 上游产物只写了文件但未回报主控。
- 备份异常且影响还原保障。

## 待确认问题

当前无待确认问题。本规则作为主控和 01-10 号助手的统一协作规则执行。

## 风险与依赖

- 自动交接必须依赖主控检查，不允许助手之间自行跳过主控直接交接。
- 自动交接不能替代关键关口的用户确认。
- 若旧线程启动提示仍引用旧目录，主控派发时必须覆盖为最新路径。
