## 关键结论

### 1. 使用原则

open-design 是参考库，不是需求来源。

k12 项目所有页面必须优先遵守：

- 已确认业务需求
- 项目工作流规则
- 信息架构与交互设计产物
- 高保真原型交付标准
- 权限边界与数据字段定义

### 2. 当前优先参考方向

领榜 SaaS 后台当前重点是用户业务管理模块，包括家长管理、教师管理。优先参考以下类型：

- dashboard / enterprise dashboard：后台首页、数据概览、管理台布局
- design-systems：表格、筛选器、状态标签、按钮、表单、弹窗、详情页组件
- kanban-board：审核、跟进、状态流转类页面
- mobile-app：家长端、教师端、小程序类页面结构

### 3. 后台页面使用方式

用于 SaaS 后台时，open-design 只能辅助以下设计决策：

- 左侧导航与顶部操作区的层级表达
- 列表页筛选区、批量操作区、分页区的信息密度
- 详情页中基础资料、需求记录、订单记录、审核记录的分区方式
- 启用/禁用、查看详情、编辑等操作按钮的视觉权重
- 状态标签、金额、订单号、手机号脱敏字段的展示规范

不得直接套用与 k12 后台业务无关的营销页、展示页、海报页或社交媒体模板。

### 4. 移动端页面使用方式

用于家长端或教师端时，open-design 可参考：

- 表单分步填写
- 入驻资料展示
- 订单状态展示
- 个人中心布局
- 信息卡片与底部操作按钮

但不得改变已确认的入驻字段、需求字段、支付字段和权限规则。

- 不依赖外部在线资源才能展示核心页面
- 不引入与业务无关的模板功能
- 修改前后保持当前导航、页面入口和模块关系一致

## 风险与依赖

- 若后续需要更新 open-design，仍可能受 GitHub 下载速度和稳定性影响。
- 外部模板存在视觉诱导风险，必须避免把模板中的功能误认为 k12 需求。
- 若直接复制模板结构，可能导致字段缺失、权限缺失、状态缺失或与 PRD 不一致。
- 高保真原型必须继续保持离线可用，不能因为参考外部资源破坏交付标准。
