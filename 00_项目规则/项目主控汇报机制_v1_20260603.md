# 项目主控汇报机制

## 背景

项目主控助手负责调度 01-10 号助手，并直接向用户汇报项目状态。此前主控已能更新台账和派发助手，但缺少“节点完成即主动汇报”的硬规则，导致用户无法及时掌握项目进度。

## 目标

建立固定协作模式：节点完成即汇报、关键关口需用户确认、汇报采用“结论 + 行动”格式。

## 输入来源

- AGENTS.md

- 00_项目规则/项目主控状态台账_v1_20260602.md
- 00_项目规则/助手向主控回报规则_v1_20260603.md
- 用户确认的协作偏好：节点完成即报、关键关口确认、结论 + 行动

## 关键结论

主控不得静默推进关键节点。每个助手完成阶段产物后，主控必须先检查产物，再向用户汇报。涉及需求范围、业务流程、原型评审和最终 PRD 的关键关口，必须等待用户确认后再进入下一阶段。

所有助手完成、阻塞、需要确认或需要回流时，必须先按统一格式回报主控。主控收集和检查后，再统一向用户汇报。

## 汇报触发规则

| 触发场景 | 主控动作 | 是否必须通知用户 |
| --- | --- | --- |
| 助手完成产物 | 检查目录、命名、通用输出要求、越界情况和风险 | 是 |
| 进入关键关口 | 汇报结论、风险、待确认问题，并等待用户确认 | 是 |
| 发现阻塞问题 | 说明阻塞原因、影响范围、建议处理动作 | 是 |
| 发现上游不一致 | 记录问题，说明是否需要回流对应助手 | 是 |
| 普通阶段衔接无阻塞 | 主控可自动派发下一助手，但必须在汇报中说明 | 是 |
| 流程资产变更 | 更新规则或台账后触发自动备份 | 可在最终汇报中说明 |

## 关键关口确认

| 关口 | 必须汇报内容 | 是否等待用户确认 |
| --- | --- | --- |
| 02 需求分析完成后 | 需求范围、MVP 候选、已确认需求、推测需求、待确认需求、暂不纳入范围 | 是 |
| 03 业务流程完成后 | 主流程、异常流程、状态流转、角色协作、权限边界 | 是 |
| 07 高保真原型评审后 | 原型问题、7 号可修问题、上游回流问题、是否进入 PRD | 是 |
| 08 最终 PRD 完成后 | PRD 完整性、未闭环问题、是否进入 09 全链路评审 | 是 |

## 自动推进边界

主控可以在以下情况下自动推进：

- 上游产物完整。
- 当前阶段不是关键关口。
- 没有新增需求范围、业务流程、权限边界、数据边界或 PRD 结论变化。
- 没有阻塞问题。
- 自动推进后仍需向用户汇报“已安排谁、下一步是什么”。

主控不得在以下情况下自动推进：

- 需求范围、MVP 或项目边界需要用户拍板。
- 主流程、异常流程、权限边界存在不确定。
- 原型评审问题涉及上游需求、流程、信息架构、UI/UX 或数据契约。
- 最终 PRD 存在未闭环关键问题。
- 用户明确要求暂停、先讨论或先确认。

## 汇报格式模板

每次汇报默认采用以下格式：

```text
结论：
当前到哪一步，是否能进入下一阶段。

已完成：
列出本轮产物和关键文件。

关键结论：
只列对决策有影响的结论。

风险/阻塞：
列出当前不确定、冲突或会影响后续工作的点。

我已安排：
说明已派发哪个助手、做什么。

需要你确认：
只列真正需要用户拍板的问题。
```

## 待确认问题

- 是否存在 `2026-06-01_20260601 k12 优坐标领榜业务讨论会.md` 的有效版本。
- 两项目各自后台是否允许共享中台数据、家长资料、老师资料、渠道标签。
- 领榜教育主业务机制是指定老师、老师抢单、老师接单还是客服/教务派单。
- 优坐标当前 MVP 主线是课程/老师展示、会员课包、B 端学习交付、排课对接，还是组合版本。

## 风险与依赖

- 若主控只更新文件不主动汇报，用户无法掌握真实进度。
- 若关键关口静默推进，后续业务流程、信息架构、原型和 PRD 可能基于未确认前提返工。
- 主控状态必须沉淀到项目文件，不能依赖聊天记忆。
