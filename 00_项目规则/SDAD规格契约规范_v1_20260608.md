# SDAD 规格契约规范

> 版本：v1 | 创建时间：20260608
> 全称：Specification-Driven Agent Delivery（规格驱动代理交付）
> 来源：对标 AgentPM 平台的 SDAD 系统，适配 K12 Codex 项目

---

## 一、核心理念

**SDAD 的 3 条铁律：**

1. **Spec 作契约**：每个阶段的输出规格文档，是下一阶段的硬输入。Agent 不猜测、不编造、不跳过。
2. **Agent 不越界**：每个 Agent 只处理本阶段的职责范围，不替上游补数据、不替下游做决策。
3. **产物即交付**：Agent 执行过程本身产出可交付的规格文档，不额外人工补写。

## 二、规格文档流转模型

```
[阶段 N 开始]
    │
    ├── 1. 读取「规格输入卡片」→ 校验上游产物完整性
    │       缺失 → 阻塞，回报主控
    │       完整 → 继续
    │
    ├── 2. Agent 按规格执行本阶段任务
    │       遵循本阶段输出清单
    │
    ├── 3. 输出本阶段产物 → 附加「规格签字卡」
    │       签字卡确认：产物完整 / 输入已消费 / 未越界
    │
    └── [阶段 N+1 开始] → 以 N 的签字卡 + 产物作为输入契约
```

## 三、规格输入卡片模板

每个阶段启动时，Agent 必须校验以下输入卡片是否完整：

| 字段 | 说明 | 校验规则 |
|------|------|---------|
| `stage_id` | 当前阶段编号 | 必须在 00-17 范围内 |
| `required_inputs` | 必需的上游产物路径 | 逐一检查文件是否存在且非空 |
| `input_spec_signed` | 上游规格签字卡是否存在 | 每个上游产物必须有对应签字卡 |
| `input_version` | 上游产物版本号 | 与上游签字卡版本号一致 |
| `blocking_issues` | 已知的阻塞问题 | 有阻塞问题不得启动 |

**规格输入校验格式：**

```markdown
## 规格输入校验

| 上游产物 | 文件路径 | 存在 | 版本 | 签字卡 | 校验结果 |
|---------|---------|------|------|--------|---------|
| 05_需求分析 | 05_需求分析/需求分析_MVP范围_v1_20260602.md | ✅ | v1 | ✅ | 通过 |
| 06_业务流程 | 06_业务流程/业务流程_主流程_v1_20260602.md | ✅ | v1 | ✅ | 通过 |
| 07_信息架构 | 07_信息架构/信息架构_模块结构_v1_20260602.md | ✅ | v1 | ✅ | 通过 |
```

## 四、规格签字卡模板

每个阶段结束时，Agent 必须在产物文件中附加签字卡：

```markdown
---

## 规格签字卡

| 项目 | 内容 |
|------|------|
| 阶段编号 | [阶段编号] |
| 产物文件 | [本文件路径] |
| 版本 | [版本号] |
| 签字时间 | [YYYYMMDD-HHmm] |
| 输入完整性 | ✅ 已读取全部上游产物 / ⚠️ 缺失xxx |
| 产物完整性 | ✅ 结构完整 / ⚠️ 缺少xxx |
| 未确认事项 | [清单或无] |
| 越界声明 | 本产物不包含[下游阶段]的决策内容 |
| 签字 | ✅ 可交付 |

**下游阶段须知：**
- [1-3 条关键注意事项]
```

## 六、契约违规处理

| 违规类型 | 处理方式 |
|---------|---------|
| 上游产物缺失 | 阻塞当前阶段，回报主控，等待补齐 |
| 上游签字卡缺失 | 要求上游 Agent 补签，阻塞当前阶段 |
| 版本不一致 | 回报主控，确认版本后重新校验 |
| Agent 越界决策 | 主控标记为「越界内容」，要求修正 |
| 产物不完整 | 退回当前 Agent，要求补全后重新签字 |

## 七、总控调度 Agent 的契约管理职责

总控调度 Agent 是 SDAD 系统的唯一执行主体，负责：

1. **启动检查**：派发任务前，校验上游产物完整性
2. **阶段调度**：按契约依赖图确定哪些 Agent 可以并行、哪些必须串行
3. **产物检查**：阶段结束后校验签字卡完整性和一致性
4. **阻塞管理**：记录阻塞项，回报用户，追踪解除
5. **版本管理**：维护全局产物版本台账
