# 助手向主控回报规则

## 背景

k12 项目采用多助手协作方式推进。阶段助手负责生成阶段产物，项目主控助手负责统一调度、检查、汇总和向用户汇报。

为避免阶段助手只写文件、不回报状态，所有助手必须在完成、阻塞、需要确认或发现上游冲突时，先向项目主控助手回报。

## 目标

建立统一回报协议，确保主控能够及时收集 01-10 号助手状态，并统一向用户汇报。

## 关键结论

所有助手完成工作、遇到阻塞、发现需要用户确认、发现上游冲突或发现职责边界外问题时，都必须向主控回报。阶段助手不得直接替用户拍板，不得绕过主控静默推进。

确认事项不得只写入阶段产物。任何助手只要在工作中发现需要用户确认的问题，必须立即在当前回合或完成回报中明确回报主控，由主控统一判断影响范围并主动告诉用户。

## 必须回报的触发场景

| 触发场景 | 回报要求 |
| --- | --- |
| 阶段产物完成 | 必须回报产物文件、关键结论、待确认问题、风险与建议下一步 |
| 上游输入缺失 | 必须说明缺少哪些输入、影响范围、建议补充来源 |
| 输出过程中发现关键冲突 | 必须说明冲突来源、影响范围、建议回流对象 |
| 发现需要用户拍板的问题 | 必须立即交给主控汇总，不得自行决定，不得只写入产物文件 |
| 发现职责边界外的问题 | 必须说明应由哪个助手处理 |
| 发现需要回流上游助手的问题 | 必须说明回流助手、问题分类、建议动作 |
| 产物无法继续推进 | 必须说明阻塞原因和解除条件 |
| 备份助手发现备份异常 | 必须报告主控 |

说明：10 号备份助手成功备份时只写备份记录，不主动打扰主控；备份异常时必须回报主控。

## 统一回报格式

所有助手向主控回报时必须使用以下格式：

```text
主控回报

当前状态：
已完成 / 进行中但阻塞 / 需要确认 / 需要回流

产物文件：
列出已生成文件路径；没有产物时说明原因。

关键结论：
列出本阶段最重要的结论。

待确认问题：
列出需要用户或上游确认的问题。

风险与依赖：
列出会影响后续阶段的问题。

建议下一步：
可进入下一阶段 / 需要用户确认 / 需要回流某助手 / 暂停等待输入。
```

## 主控处理要求

主控收到助手回报后必须：

- 检查产物目录和文件命名。
- 检查是否包含通用输出要求。
- 检查是否越界。
- 判断是否进入关键关口。
- 判断是否需要用户确认。
- 对需要用户确认的问题判断是否阻塞当前阶段、是否可带问题推进、是否需要回流。
- 更新主控状态台账。
- 必要时触发 10 号备份助手。
- 用“结论 + 行动”格式向用户汇报。

如果存在用户确认项，主控必须主动告诉用户；如果不存在阻塞确认项，节点汇报中也必须说明“当前无阻塞确认项”。

## 禁止事项

- 阶段助手不得只写文件、不回报主控。
- 阶段助手不得把待确认事项只藏在阶段产物里。
- 阶段助手不得直接替用户拍板。
- 阶段助手不得把待确认内容写成已确认结论。
- 阶段助手不得绕过主控直接推进下游。
- 阶段助手不得把职责边界外的问题自行处理成正式结论。

## 待确认问题

- 当前无新增待确认问题。本规则作为所有助手的统一协作规则执行。

## 风险与依赖

- 如果助手不回报主控，主控无法及时向用户同步进度。
- 如果助手自行处理需要用户确认的问题，后续业务流程、原型和 PRD 可能返工。
