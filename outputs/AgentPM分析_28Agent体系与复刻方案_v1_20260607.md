## 二、7 大阶段与 AI 全程驱动机制

### 2.1 阶段总览

```
[需求规划] → [架构设计] → [原型设计] → [代码开发] → [测试验证] → [部署上线] → [项目交付]
    ↑                                                                              │
    └──────────────────────── 规格文档作为阶段契约流转 ───────────────────────────────┘
```

### 2.2 每阶段 AI 驱动方式

| 阶段 | AI 驱动方式 | 输入 | AI 输出 |
|------|------------|------|---------|
| **需求规划** | AI 自动进行需求研讨、头脑风暴、结构化 | 一句话/原始需求描述 | 需求规格说明书 (SRS)、优先级矩阵、路线图 |
| **架构设计** | AI 按规格文档生成概要设计、详细设计 | SRS | 架构设计文档、数据库设计、API 规范 |
| **原型设计** | Page Generator Agent 按需求说明书生成可交互原型 | 需求说明书 + 架构设计 | 高保真 HTML 原型（可运行/可跳转/可交互） |
| **代码开发** | AI 编码 Agent 生成前后端代码 | 原型 + API 设计 | 前端代码、后端代码、SQL 脚本 |
| **测试验证** | AI 按需求说明书自动生成四类测试用例 | SRS + 代码 | 功能/异常/性能/权限测试用例 |
| **部署上线** | AI 配置 CI/CD、一键部署到专属子域名 | 构建产物 | 部署配置、环境文档 |
| **项目交付** | AI 自动生成操作手册、Word 导出 | 全部上游产物 | 用户手册、交付包 |

### 2.3 SDAD 机制核心原则

1. **Spec 作契约**：每阶段输出的规格文档是下一阶段的硬输入，Agent 不猜测、不越界
2. **自然语言驱动**：产品经理全程用自然语言交互，无需编写代码
3. **标准化产物**：所有产物格式统一、可评审、可导出、可直接使用
4. **多 Agent 协作**：26+ Agent 按阶段分工，通过规格文档串联而非记忆传递
5. **一键式操作**：从创建项目到生成原型到部署，均为引导式一键完成

## 四、K12 项目 1:1 复刻方案

### 4.1 现状对比

| 维度 | AgentPM | K12 Codex 项目 | 差距分析 |
|------|---------|---------------|---------|
| Agent 数量 | 26+ | 11 | K12 缺少 15+ Agent |
| 阶段数 | 7 | 12 | K12 更细粒度，但缺少前/后端阶段 |

### 4.3 复刻方案选择

#### 方案 A：最小复刻（12 Agent）

**思路**：在 K12 现有 11 Agent 基础上，新增 1 个「总控调度 Agent（强化版）」，引入 SDAD 机制但保持现有流程。

**新增 Agent**：
- 00-强化: 总控调度 Agent（引入规格契约机制）

**改动范围**：最小。仅强化 00 主控助手的规格文档管理能力。

**优点**：零风险、即插即用。
**缺点**：很多 AgentPM 能力无法覆盖。

#### 方案 C：差异化复刻（K12 定制版，推荐）

**思路**：保留 K12 现有 12 阶段 + 11 Agent 架构，引入 SDAD 的「规格契约」机制作为核心增强，选择性补齐缺失的关键 Agent。

**核心改造**（3 项）：

**改造 1：强化总控调度 Agent（00-NEW）**
- 新增能力：规格文档版本管理、阶段间契约校验、产物完整性检查
- 在每个阶段开始前，自动注入上游阶段的规格文档
- 在每个阶段结束后，校验产物是否符合规格契约

**改造 2：在每个阶段引入「规格输入/输出签字」机制**
```
阶段 N 开始 → 读取阶段 N-1 的规格文档（强制性）
阶段 N 执行 → AI Agent 按规格执行
阶段 N 结束 → 输出阶段 N 的产物 + 规格签字
阶段 N+1 → 必须读取 N 的签字产物才能启动
```

**改造 3：补齐 5 个关键 Agent**

| 编号 | Agent 名称 | 理由 |
|------|-----------|------|
| 01-NEW | 市场调研 Agent | 补齐前期研究能力，当前会议流程缺失 |
| 02-NEW | 竞品分析 Agent | 领榜/优坐标需要竞品对标 |
| 03-NEW | 用户画像 Agent | C 端产品必需 |
| 06-NEW | 优先级排序 Agent | 需求价值评估缺失 |
| 09-NEW | 系统架构 Agent | 架构设计独立于业务流程 |

**目录扩展**：
```

```

## 五、技术实现：WorkBuddy 中的 Agent 复刻方式

### 5.1 AgentPM Agent vs WorkBuddy 能力对照

| AgentPM Agent 能力 | WorkBuddy 实现方式 |
|--------------------|--------------------|
| 专业 Agent（专用 Prompt） | 创建专用 Expert / Agent 对话线程，注入固定系统指令 |
| 规格文档流转 | 文件系统读写：每个 Agent 强制读取上游产物 |
| 多 Agent 协作 | 顺序调度：一个对话线程完成后，下一个线程加载产物 |
| 一键生成（原型/文档） | Web/Code Generation 能力已内置 |
| Word 导出 | docx 技能 |
| 原型 HTML 输出 | 原生 HTML 生成 + 本地预览 |
| 数据库设计 | 结构化 Markdown 输出 |

### 5.2 实施关键点

1. **Agent 系统指令标准化**：每个 Agent 的 Prompt 必须包含「输入契约要求」和「输出规格要求」两个固定模块
2. **产物格式强约束**：所有 Agent 产物采用统一的 Markdown 模板，包含固定字段（背景/目标/输入来源/关键结论/待确认/风险/下一步/规格签字）
3. **不可跳过规则**：00 总控 Agent 启动每个子 Agent 前，必须校验上游产物是否存在且包含「规格签字」字段

## 六、关键洞察与建议

### 6.1 K12 项目的差异化优势

- **12 阶段更细粒度**：比 AgentPM 的 7 阶段更适合需要频繁会议对齐的协作场景
- **教育领域专精**：AgentPM 是通用平台，K12 可以在教育业务上有更深的 Agent 适应
- **已有可交互原型能力**：K12 已经做到 HTML 可交互原型（AgentPM 的核心卖点之一）

### 6.2 需要警惕的坑

- **Agent 数量膨胀**：AgentPM 的 26+ Agent 中，原型设计阶段就有 6 个 Agent（12-17），这在工作流中可能导致过度拆分
- **规格契约的刚性**：SDAD 适合瀑布式开发，敏捷迭代时需要灵活处理规格文档的版本管理
- **AI 生成代码的价值**：AgentPM 的代码/测试/部署阶段对 K12 场景价值有限，不建议复刻

### 6.3 决策建议

| 场景 | 推荐方案 |
|------|---------|
| 快速验证 SDAD 思路 | 方案 A（最小复刻，1 天） |
| 完整对标 AgentPM | 方案 B（完整复刻，1 周） |
| **平衡投入产出** | **方案 C（K12 定制版，3 天）← 推荐** |
