# k12 项目 Codex 协作规则

## 文件命名

所有文件命名必须按照：

文件主题_调整内容_版本号_创建时间

创建时间格式为年月日，例如：20260602。

示例：

- 会议记录_原始转写_v1_20260602.md
- 需求分析_MVP范围_v1_20260602.md
- 高保真原型_用户管理_v1_20260602.html

## 多对话协作方式

不同对话负责不同阶段，但必须通过项目文件串联，不依赖其他对话的聊天记忆。

每个对话开始工作前必须先读取：

1. 根目录 AGENTS.md
2. 00_项目规则/项目总览_工作流程_v1_20260602.md
3. 13_任务卡片/ 下与自己角色相关的任务卡片
4. 当前任务所需的上游阶段产物

每个对话结束时必须产出结构化文件，写入对应阶段目录。

## 上下文交接

当主控助手或任一阶段助手上下文接近满、旧线程系统错误或用户要求新助手接替时，必须执行：

- `00_项目规则/上下文交接规则_v1_20260606.md`
- `00_项目规则/上下文交接模板_v1_20260606.md`

新接替助手线程必须按以下格式命名：

```text
原有助手名称 V版本号 交接日期
```

示例：

- `大统领 V0.1 20260606`
- `07 - 高保真原型助手 V0.1 20260606`

版本号从 `V0.1` 开始，同一助手再次交接时递增；交接日期精确到日，格式为 `YYYYMMDD`。

## 工作流程

项目按照以下顺序推进：

1. 会议录音转文字
2. 会议内容整理
3. 需求提取
4. 需求澄清
5. 需求分析
6. 业务流程设计
7. 信息架构设计
8. 交互设计
9. 前后端数据交互设计
10. 高保真原型设计
11. 详细需求文档编写
12. 评审与修订

## 通用输出要求

所有阶段性产物必须包含：

- 背景
- 目标
- 输入来源
- 关键结论
- 待确认问题
- 风险与依赖
- 下一步动作

## 需求分析规则

必须明确区分：

- 已确认需求
- 推测需求
- 待确认需求
- 暂不纳入范围

不得把会议里的个人观点直接当成最终需求。

## 业务流程规则

业务流程必须覆盖：

- 主流程
- 异常流程
- 角色协作
- 状态流转
- 权限边界
- 业务规则
- 关键决策点

## 信息架构规则

信息架构必须覆盖：

- 模块结构
- 页面结构
- 导航关系
- 页面入口
- 页面出口
- 页面状态
- 页面之间的上下游关系

## 交互设计规则

交互设计必须覆盖：

- 页面流程
- 操作路径
- 组件状态
- 表单规则
- 跳转逻辑
- 反馈机制
- 异常交互
- 权限差异下的交互变化

## 前后端交互规则

涉及数据交互时必须说明：

- 接口场景
- 请求字段
- 返回字段
- 状态码或错误状态
- 权限校验
- 数据校验规则
- 前端展示规则
- 后端处理规则

## 原型设计规则

高保真原型必须采用可交互 HTML 作为交付标准，并覆盖：

- 核心流程
- 异常流程
- 空状态
- 加载状态
- 错误状态
- 权限差异
- 数据刷新逻辑

## 评审规则

评审时优先检查：

- 需求是否来源清晰
- 是否存在未确认假设
- 流程是否断裂
- 信息架构是否支撑业务流程
- 交互设计是否覆盖关键操作路径
- 异常状态是否缺失
- 权限边界是否清楚
- 字段和接口是否支撑页面交互
- PRD、流程、交互、原型之间是否一致
